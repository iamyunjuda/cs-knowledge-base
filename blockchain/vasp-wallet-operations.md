# VASP 지갑 운영 — 거래소 지갑 아키텍처, Travel Rule, 규제 준수

## 핵심 정리

### 1. VASP란?

```
[VASP = Virtual Asset Service Provider (가상자산사업자)]

  FATF(국제자금세탁방지기구) 정의:
  가상자산의 교환, 이전, 보관, 관리 등을 업으로 수행하는 사업자

  VASP 유형:
  ┌──────────────────┬────────────────────────────────────────┐
  │ 유형              │ 예시                                    │
  ├──────────────────┼────────────────────────────────────────┤
  │ 중앙화 거래소(CEX) │ Upbit, Bithumb, Binance, Coinbase      │
  │ 커스터디 서비스    │ BitGo, Fireblocks, Anchorage            │
  │ 결제 서비스       │ BitPay, Circle                          │
  │ 가상자산 ATM      │ Bitcoin ATM 운영사                       │
  │ DeFi 프론트엔드   │ 일부 국가에서 VASP로 분류 논의 중          │
  └──────────────────┴────────────────────────────────────────┘

  한국 특정금융정보법(특금법) 기준 VASP:
  - 가상자산 매매/교환
  - 가상자산 이전(전송)
  - 가상자산 보관/관리
  - 가상자산 매매/교환 중개/알선
  → 신고 의무, 실명계좌 확보, 정보보호관리체계(ISMS) 인증 필수
```

### 2. 거래소 지갑 아키텍처

```
[거래소 전체 지갑 구조]

  ┌──────────────────────────────────────────────────────┐
  │                    사용자 계층                         │
  │  ┌──────────┐ ┌──────────┐ ┌──────────┐             │
  │  │ User A   │ │ User B   │ │ User C   │ ...          │
  │  │ 입금주소  │ │ 입금주소  │ │ 입금주소  │             │
  │  │ 0xAAA    │ │ 0xBBB    │ │ 0xCCC    │             │
  │  └────┬─────┘ └────┬─────┘ └────┬─────┘             │
  └───────┼────────────┼────────────┼────────────────────┘
          │ Sweep      │ Sweep      │ Sweep
          ▼            ▼            ▼
  ┌──────────────────────────────────────────────────────┐
  │                    수집 (Sweep) 계층                   │
  │  입금 감지 → 사용자 잔액 갱신 → 자산을 통합 지갑으로 이동  │
  │                        │                              │
  └────────────────────────┼──────────────────────────────┘
                           ▼
  ┌──────────────────────────────────────────────────────┐
  │                    통합 지갑 (Omnibus Wallet)           │
  │  ┌──────────────────────────────────────────────┐    │
  │  │ Hot Wallet: 일일 출금 처리용 (전체의 ~5%)      │    │
  │  │ → KMS 보호, 자동 서명, 일일 한도 설정          │    │
  │  └──────────────────┬───────────────────────────┘    │
  │                     │ 부족 시 충전                     │
  │  ┌──────────────────▼───────────────────────────┐    │
  │  │ Warm Wallet: Hot 충전용 (전체의 ~10%)          │    │
  │  │ → KMS 보호, 반자동 승인                       │    │
  │  └──────────────────┬───────────────────────────┘    │
  │                     │ 부족 시 충전 (수동)              │
  │  ┌──────────────────▼───────────────────────────┐    │
  │  │ Cold Wallet: 대량 보관 (전체의 ~85%)           │    │
  │  │ → HSM/MPC, 멀티시그, Air-gapped              │    │
  │  └──────────────────────────────────────────────┘    │
  └──────────────────────────────────────────────────────┘

[Sweep(수집) 프로세스]

  1. 입금 감지:
     이벤트 리스너가 사용자 입금 주소로의 Transfer 이벤트 감지
     → DB에 입금 기록 (status: PENDING)

  2. Confirmation 대기:
     12+ confirmations 후 → status: CONFIRMED
     사용자 잔액에 반영

  3. Sweep 실행:
     사용자 입금 주소 → Hot Wallet으로 자산 이동
     → 개별 주소에 자산이 분산되면 관리 어려움
     → 통합 지갑으로 모아야 출금 처리 효율적

  주의사항:
  - Sweep에도 가스비 필요 → 소액 입금은 가스비 > 입금액일 수 있음
  - ERC-20 토큰 Sweep 시 해당 주소에 ETH(가스비)가 있어야 함
    → 가스비용 ETH를 먼저 보내야 함 (2개 Tx 필요)
  - 대안: Sweep 없이 사용자 주소에서 직접 출금 (복잡하지만 가스비 절약)
```

### 3. 입출금 처리 플로우

```
[입금 처리]

  외부 지갑 → Transfer → 사용자 입금 주소 (0xAAA)
    │
    ▼
  이벤트 리스너: Transfer 이벤트 감지
    │
    ▼
  입금 기록 생성 (status: DETECTED, confirmations: 1)
    │
    ▼
  Confirmation 모니터링 (블록마다 확인)
    │
    ├── 12 conf: status → CONFIRMED
    │   └── 사용자 잔액 반영 (available += amount)
    │   └── 알림: "입금 완료"
    │
    └── Reorg 감지 시: status → REORGED
        └── 잔액 롤백
        └── 알림: "입금 취소 (네트워크 재조직)"

[출금 처리]

  사용자 출금 요청
    │
    ▼
  검증 단계:
  ├── 잔액 충분? (available >= amount + fee)
  ├── 출금 주소 유효? (체크섬, 블랙리스트)
  ├── 일일 한도 초과?
  ├── AML/Travel Rule 충족?
  └── 2FA 인증 완료?
    │
    ▼
  잔액 Lock (available -= amount, locked += amount)
    │
    ▼
  출금 큐에 추가
    │
    ▼
  출금 프로세서:
  ├── Hot Wallet 잔액 확인
  ├── Gas 추정 및 Tx 구성
  ├── Policy Engine 승인 (자동/수동)
  └── KMS/HSM에서 서명 → 블록체인 전송
    │
    ▼
  Tx 모니터링:
  ├── CONFIRMED → locked -= amount, 완료 알림
  ├── REVERTED → locked -= amount, available += amount, 재시도/알림
  └── LOST → 복구 프로세스 실행

[배치 출금 최적화]

  개별 출금은 가스비 비효율적
  → 여러 출금을 모아 한 번에 처리

  방법 1: 멀티 전송 컨트랙트
    하나의 Tx로 여러 주소에 동시 전송
    → 가스비 절약 (~30-50%)

  방법 2: 주기적 배치
    매 10분마다 대기 중인 출금을 모아서 처리
    → 사용자는 약간의 지연, 서비스는 가스비 절약
```

### 4. Travel Rule

```
[Travel Rule = 가상자산 이전 시 송수신인 정보를 교환해야 하는 규정]

  FATF 권고안 16 (Recommendation 16):
  일정 금액 이상의 가상자산 이전 시
  송금인과 수취인의 신원 정보를 상대방 VASP에 전달해야 함

  한국 기준: 100만원 이상 이전 시 적용 (2022.03~)
  국제 기준: FATF는 $1,000/€1,000 이상 권고 (국가마다 다름)

[Travel Rule 교환 정보]

  송금인 정보:
  - 이름 (실명)
  - 지갑 주소
  - VASP 식별 정보 (업체명, 국가)

  수취인 정보:
  - 이름 (실명)
  - 지갑 주소
  - 수취 VASP 식별 정보

[Travel Rule 프로토콜/솔루션]

  ┌──────────────┬────────────────────────────────────────┐
  │ 솔루션        │ 설명                                    │
  ├──────────────┼────────────────────────────────────────┤
  │ CODE (한국)   │ 한국 VASP간 Travel Rule 프로토콜         │
  │              │ 업비트, 빗썸 등 참여                     │
  │ VerifyVASP   │ 한국 중심 Travel Rule 솔루션             │
  │ TRUST        │ Coinbase 주도, 미국 중심                 │
  │ Sygna Bridge │ 아시아 중심 Travel Rule 네트워크          │
  │ Notabene     │ 글로벌 Travel Rule 플랫폼               │
  │ Chainalysis  │ 블록체인 분석 + Travel Rule 지원         │
  └──────────────┴────────────────────────────────────────┘

[Travel Rule 구현 흐름]

  VASP A (송금) → VASP B (수취)

  1. VASP A: 송금 요청 접수
  2. VASP A: 수취 주소의 소속 VASP 확인 (주소 귀속 확인)
  3. VASP A → Travel Rule 네트워크 → VASP B:
     송금인/수취인 정보 전달
  4. VASP B: 수취인 정보 검증 (이름, 주소 일치 확인)
  5. VASP B → Travel Rule 네트워크 → VASP A:
     승인 또는 거절
  6. 승인 시: VASP A가 온체인 전송 실행
  7. 거절 시: 송금 불가, 사용자에게 사유 안내

[Unhosted Wallet (개인 지갑) 이슈]

  문제:
  수취 주소가 MetaMask 같은 개인 지갑이면
  → 상대방 VASP가 없음 → Travel Rule 적용 불가?

  각국 대응:
  - 한국: 외부 지갑 전송 시 소유자 확인 절차 필요
         (주소 소유 증명: 소액 테스트 전송 등)
  - EU(MiCA): €1,000 이상 외부 지갑 전송 시 검증 강화
  - 일부 국가: 외부 지갑 전송 자체를 제한
```

### 5. AML/KYC 연동

```
[AML (Anti-Money Laundering) 체크]

  블록체인 특유의 AML 작업:
  1. 주소 스크리닝: 제재 대상 주소인지 확인
     → OFAC SDN List, Chainalysis 등 활용
     → 토네이도 캐시(Tornado Cash) 관련 주소 차단 등

  2. 트랜잭션 모니터링:
     → 의심 거래 패턴 감지 (구조화, 레이어링)
     → 다크넷, 믹서, 제재 대상 프로토콜 연관 확인

  3. 리스크 스코어링:
     → 주소별 리스크 점수 산출
     → 고위험 주소와 거래 이력이 있는 주소도 주의

[KYC (Know Your Customer)]

  거래소 필수 절차:
  Level 1: 이메일/전화번호 인증
  Level 2: 실명 인증 (신분증 + 실명계좌)
  Level 3: 추가 서류 (소득 증빙, 자금 출처 등)

  한국 특금법 요구사항:
  - 실명 확인 입출금 계정(은행 실명계좌 연동)
  - 고객확인의무(CDD)
  - 강화된 고객확인(EDD) — 고위험 고객
  - 의심거래보고(STR) — 금융정보분석원(FIU)

[주소 귀속 확인 (Address Attribution)]

  입금 시:
  → 사용자별 고유 입금 주소 할당 → 누구의 입금인지 자동 식별

  출금 시:
  → 사용자가 입력한 출금 주소의 소속 확인 필요
  → Chainalysis, Elliptic 등의 API로 확인:
     "이 주소는 Binance 소속입니다" → VASP간 Travel Rule 적용
     "이 주소는 식별 불가" → 개인 지갑 추정 → 추가 검증
```

### 6. 거래소 백엔드 시스템 구성

```
[전체 시스템 구성도]

  ┌─────────────────────────────────────────────────────────┐
  │                     API Gateway                          │
  │  인증, Rate Limit, 라우팅                                │
  └───────────┬───────────────────────────────┬──────────────┘
              │                               │
  ┌───────────▼──────────┐      ┌─────────────▼────────────┐
  │    Trading Engine     │      │    Wallet Service         │
  │  주문 매칭, 체결       │      │  입출금, 잔액 관리         │
  │  오더북 관리           │      │  Sweep, 충전 자동화       │
  └───────────────────────┘      └────────────┬─────────────┘
                                               │
                                 ┌─────────────▼─────────────┐
                                 │    Blockchain Gateway       │
                                 │  Tx 구성, 서명, 전송        │
                                 │  이벤트 리스닝, Nonce 관리  │
                                 │  RPC 로드밸런싱             │
                                 └─────────────┬─────────────┘
                                               │
                                 ┌─────────────▼─────────────┐
                                 │    Signing Service          │
                                 │  KMS/HSM/MPC 연동           │
                                 │  Policy Engine              │
                                 │  감사 로깅                  │
                                 └───────────────────────────┘

  ┌───────────────────────┐      ┌───────────────────────────┐
  │    Compliance Service  │      │    Monitoring Service      │
  │  AML/KYC 검증          │      │  Tx 상태 추적             │
  │  Travel Rule 처리      │      │  지갑 잔액 모니터링        │
  │  STR(의심거래) 보고     │      │  알림 / 대시보드           │
  └───────────────────────┘      └───────────────────────────┘

[서비스 간 통신]

  Trading Engine ↔ Wallet Service:
    체결 후 잔액 업데이트, 출금 요청
    → 동기 (gRPC) + 비동기 (Kafka)

  Wallet Service → Blockchain Gateway:
    Tx 전송 요청
    → Kafka (Outbox 패턴, 최소 1회 전달 보장)

  Blockchain Gateway → Wallet Service:
    Tx 확정/실패 알림, 입금 감지
    → Kafka (이벤트 스트리밍)

  모든 서비스 → Compliance Service:
    출금 전 AML 체크, Travel Rule 데이터 교환
    → 동기 (gRPC, 출금 차단을 위해)
```

## 헷갈렸던 포인트

### Q1: 개인 지갑 vs 거래소 지갑, 입금 주소의 차이?

```
[개인 지갑 (Non-custodial)]
  MetaMask, Ledger 등
  → 사용자가 개인키를 직접 소유
  → 주소 1개가 곧 사용자의 "계좌"

[거래소 지갑 (Custodial)]
  Upbit, Binance 등
  → 거래소가 개인키를 보관 (수탁)
  → 사용자별 "입금 주소"를 할당하지만
     실제 자산은 거래소의 통합 지갑에 모여 있음

  사용자 A의 "잔액 2 ETH"는:
  → 사용자 A의 입금 주소에 2 ETH가 있는 게 아님
  → 거래소 DB에 "A: 2 ETH"로 기록되어 있을 뿐
  → 실제 ETH는 거래소의 Hot/Cold Wallet에 통합 보관

  "Not your keys, not your coins"
  → 거래소에 맡긴 자산은 거래소를 신뢰해야 함
  → FTX 사태가 대표적인 리스크 사례
```

### Q2: Sweep 비용이 입금액보다 크면?

```
[더스트(Dust) 문제]

  사용자가 0.001 ETH를 입금
  Sweep Tx 가스비: 0.002 ETH
  → Sweep하면 손해!

  대응 전략:
  1. 최소 입금액 설정: "0.01 ETH 미만 입금 불가"
  2. 더스트 집계: 소액을 모아서 일정 금액이 되면 한번에 Sweep
  3. 가스비 낮을 때 Sweep: 가스비 모니터링 → 저렴할 때 배치 Sweep
  4. 사용자 부담: 출금 수수료에 Sweep 비용 포함

  ERC-20의 경우 더 복잡:
  → 토큰 Sweep에 ETH(가스비)가 필요
  → ETH를 먼저 보내고 → 토큰을 Sweep
  → 총 2개의 Tx → 비용 2배
```

### Q3: 거래소가 해킹당하면 사용자 자산은?

```
[거래소 보안 사고 대응]

  Hot Wallet 해킹 시:
  → 전체 자산의 ~5%만 영향 (나머지 Cold)
  → 서비스 일시 중단, 입출금 차단
  → 남은 자산으로 사용자 보상 또는 보험 처리

  Cold Wallet 해킹 시:
  → 이론적으로 MPC/멀티시그 돌파 필요
  → 현실적으로는 내부자 공모 케이스가 대부분
  → 발생 시 대규모 손실 → 거래소 파산 가능

  제도적 안전장치 (한국):
  - 준비금 증명 (Proof of Reserves) 의무화 논의 중
  - 이용자 자산과 회사 자산 분리 보관
  - 보험 가입 의무 (일부)
  - ISMS 인증 필수
```
