# 블록체인 분산 네트워크 — 중앙 서버 없이 20만 노드가 연결되는 원리

## 핵심 정리

### 근본적인 질문: 중앙 서버가 없는데 어떻게 서로를 찾지?

블록체인 네트워크는 **P2P(Peer-to-Peer)** 방식으로 동작한다. 카카오톡처럼 중앙 서버가 메시지를 중계하는 게 아니라, 각 노드가 직접 다른 노드들과 연결하여 데이터를 주고받는다.

```
[ 중앙 서버 방식 (Web2) ]          [ P2P 방식 (블록체인) ]

    ┌──────────┐                   A ──── B
    │  Server  │                   │╲   ╱│
    └────┬─────┘                   │ ╲ ╱ │
   ╱     │     ╲                   │  ╳  │
  A      B      C                  │ ╱ ╲ │
                                   C ──── D
  모든 통신이 서버를 거침           노드끼리 직접 연결
```

### 1단계: 최초 연결 — 부트스트랩 노드 (Bootstrap Nodes)

가장 핵심적인 질문: **"네트워크에 처음 참여할 때, 아무도 모르는데 누구에게 연결하지?"**

답은 **부트스트랩 노드(Bootstrap Node)** 또는 **시드 노드(Seed Node)**다.

```
[ 새 노드가 네트워크에 참여하는 과정 ]

1) 클라이언트 소프트웨어(예: Geth)에 부트스트랩 노드 주소가 하드코딩되어 있음

   // Geth 소스코드에 실제로 박혀 있는 부트스트랩 노드들
   var MainnetBootnodes = []string{
       "enode://d860a01f...@18.138.108.67:30303",
       "enode://22a8232c...@3.209.45.79:30303",
       "enode://2b252ab6...@65.108.70.101:30303",
       ...
   }

2) 새 노드 → 부트스트랩 노드에 접속
3) 부트스트랩 노드가 "내가 알고 있는 다른 노드 목록"을 알려줌
4) 새 노드는 그 목록의 노드들에게 다시 연결 → 그 노드들도 자기가 아는 목록 공유
5) 이렇게 점점 네트워크를 알아감 (눈덩이처럼)
```

> **비유**: 새로 전학 온 학생이 담임선생님(부트스트랩 노드)한테 "반 친구들 소개해줘"라고 하면, 선생님이 몇 명을 소개해주고, 그 친구들이 또 다른 친구를 소개해주는 것과 같다.

**부트스트랩 노드가 죽으면?**
- 부트스트랩 노드는 여러 개 존재 (이더리움은 약 6~8개)
- 이미 연결된 노드들은 부트스트랩 노드 없이도 계속 동작
- 한번 네트워크에 참여하면, 이전에 연결했던 노드 목록을 로컬에 캐싱
- 다음에 재시작할 때는 캐싱된 목록으로 바로 연결 시도

### 2단계: 노드 탐색 — Kademlia DHT 프로토콜

부트스트랩 노드로 초기 연결을 했으면, 이제 **"네트워크에 있는 다른 노드들을 효율적으로 찾는 방법"**이 필요하다.

이더리움은 **Kademlia** 기반의 DHT(Distributed Hash Table) 프로토콜을 사용한다.

```
[ Kademlia 노드 탐색 원리 ]

1. 모든 노드는 고유한 Node ID를 가짐 (256비트)
   - Node ID = 공개키의 Keccak-256 해시

2. 노드 간 "거리"를 XOR 연산으로 계산
   - XOR 거리 = Node ID_A ⊕ Node ID_B
   - 물리적 거리가 아님! 논리적 거리

3. 각 노드는 "K-Bucket" 라우팅 테이블을 유지
   ┌─────────────────────────────────────────┐
   │ K-Bucket (k=16)                         │
   │                                         │
   │ 거리 2^0 ~ 2^1   : [node1, node2, ...]  │  ← 가까운 노드 많이 알고 있음
   │ 거리 2^1 ~ 2^2   : [node3, node4, ...]  │
   │ 거리 2^2 ~ 2^3   : [node5, ...]         │
   │ ...                                     │
   │ 거리 2^255 ~ 2^256: [nodeN]              │  ← 먼 노드는 적게 알고 있음
   └─────────────────────────────────────────┘

4. 특정 노드를 찾을 때: 재귀적으로 가장 가까운 노드에게 물어봄
   "ID가 0xABCD...인 노드 알아?"
   → "모르지만, 그쪽에 가까운 노드 3개 알려줄게"
   → 그 3개에게 다시 물어봄
   → 반복하면 O(log N)만에 찾음!
```

**핵심 포인트**: 20만 노드가 있어도, 어떤 노드든 **최대 17번 정도**(log₂ 200,000 ≈ 17)만 질문하면 찾을 수 있다.

### 3단계: 실제 연결 — devp2p / libp2p 프로토콜

노드를 발견했으면, 이제 **실제로 데이터를 주고받는 통신 프로토콜**이 필요하다.

```
[ 이더리움 네트워크 프로토콜 스택 ]

┌──────────────────────────────────┐
│     ETH Protocol (eth/68)        │  ← 블록, 트랜잭션 교환
├──────────────────────────────────┤
│     SNAP Protocol                │  ← 빠른 상태 동기화
├──────────────────────────────────┤
│     RLPx Transport               │  ← 암호화된 통신 채널
│     (ECIES 암호화 + 프레이밍)     │
├──────────────────────────────────┤
│     Node Discovery v4/v5         │  ← Kademlia 기반 노드 탐색
│     (UDP)                        │
├──────────────────────────────────┤
│     TCP / UDP                    │  ← 전송 계층
└──────────────────────────────────┘
```

**연결 과정**:
```
1. 노드 탐색 (UDP 30303 포트)
   A → B: PING (나 살아있어)
   B → A: PONG (나도 살아있어)
   A → B: FIND_NODE (이 ID 근처 노드 알려줘)
   B → A: NEIGHBORS (이런 노드들 있어)

2. 실제 연결 수립 (TCP 30303 포트)
   A → B: 암호화 핸드셰이크 (ECIES)
   A ↔ B: RLPx 프레임으로 메시지 교환

3. 데이터 교환 (ETH 프로토콜)
   - 새 블록 전파
   - 새 트랜잭션 전파
   - 블록체인 동기화
```

### 4단계: 블록/트랜잭션 전파 — 가십 프로토콜 (Gossip Protocol)

20만 노드 전체에 새 트랜잭션이나 블록을 어떻게 빠르게 퍼뜨릴까?

**Gossip Protocol** (소문 퍼뜨리듯 전파)을 사용한다.

```
[ 가십 프로토콜로 트랜잭션 전파 ]

시간 0: 유저가 트랜잭션 생성 → 연결된 노드 A에 전송

시간 1: A가 자기가 연결된 피어들에게 전파
         A → B, C, D  (보통 √N개의 피어에게 전체 데이터 전송)

시간 2: B, C, D가 각자의 피어들에게 전파
         B → E, F
         C → G, H
         D → I, J

시간 3: E, F, G, H, I, J → ...

         ┌─ 시간 0: 1개 노드
         ├─ 시간 1: ~15개 노드
         ├─ 시간 2: ~200개 노드
         ├─ 시간 3: ~3,000개 노드
         ├─ 시간 4: ~45,000개 노드
         └─ 시간 5: ~200,000개 노드 (전체 도달!)

→ 보통 6~12초 내에 네트워크 전체에 전파됨
```

**이더리움의 최적화된 전파 방식**:
```
1. 연결된 피어 중 √N개에게 → 전체 블록/Tx 전송
2. 나머지 피어들에게 → 해시만 전송 (announcement)
3. 해시를 받은 노드가 아직 그 데이터를 모르면 → 전체 데이터 요청

이렇게 하면 대역폭을 대폭 절약하면서도 빠르게 전파 가능
```

### 5단계: 각 노드는 전체 데이터를 저장한다

```
[ 각 풀노드가 저장하는 데이터 ]

풀노드 하나 ≈ 1TB+ 저장 공간 필요 (이더리움 기준)

┌───────────────────────────────────────┐
│ 블록 데이터                             │
│  - 블록 헤더 (해시, 타임스탬프, 난이도)    │
│  - 트랜잭션 목록                        │
│  - 영수증 (Receipt)                     │
├───────────────────────────────────────┤
│ 상태 데이터 (State Trie)                │
│  - 모든 계정의 잔액                      │
│  - 컨트랙트 코드                        │
│  - 컨트랙트 스토리지                     │
├───────────────────────────────────────┤
│ 피어 정보                               │
│  - 연결했던 노드 목록 (캐싱)              │
│  - 라우팅 테이블 (K-Bucket)             │
└───────────────────────────────────────┘

→ 모든 풀노드가 동일한 데이터를 각각 독립적으로 저장
→ 이것이 탈중앙화의 핵심: 한 곳이 죽어도 데이터가 사라지지 않음
```

### 전체 흐름 요약

```
[ 노드 하나가 네트워크에 참여해서 블록을 받기까지 ]

1. Geth 실행
   └→ 하드코딩된 부트스트랩 노드에 접속

2. 부트스트랩 노드에게 피어 목록 받음
   └→ Kademlia DHT로 더 많은 노드 탐색

3. 발견한 노드들과 TCP 연결 수립
   └→ RLPx 암호화 핸드셰이크
   └→ 보통 25~50개 피어와 연결 유지

4. ETH 프로토콜로 블록체인 동기화
   └→ 제네시스 블록부터 최신 블록까지 다운로드
   └→ 각 블록을 독립적으로 검증

5. 동기화 완료 후 실시간 참여
   └→ 새 블록/Tx를 가십 프로토콜로 수신
   └→ 검증 후 자기 DB에 저장
   └→ 다른 피어들에게 다시 전파
```

### 비트코인 vs 이더리움 네트워크 비교

| 항목 | 비트코인 | 이더리움 |
|------|----------|----------|
| 노드 탐색 | DNS Seed + addr 메시지 | Kademlia DHT (discv4/v5) |
| 전송 프로토콜 | 자체 프로토콜 (TCP 8333) | devp2p / RLPx (TCP 30303) |
| 기본 피어 수 | 8 outbound + 117 inbound | 25 outbound + 25 inbound |
| 블록 전파 | Compact Block Relay (BIP 152) | eth/68 + SNAP |
| 전파 시간 | ~2~5초 (네트워크 전체) | ~1~3초 (네트워크 전체) |
| 노드 수 (2024) | ~15,000~20,000 | ~6,000~10,000 (실행 노드) |
| 체인 크기 | ~600GB | ~1TB+ (pruned ~500GB) |

### 비트코인의 DNS Seed 방식

비트코인은 부트스트랩 노드 대신 **DNS Seed** 방식을 주로 사용한다:

```
1. 비트코인 클라이언트가 DNS 쿼리 전송
   dig seed.bitcoin.sipa.be    → 여러 노드의 IP 주소 반환
   dig dnsseed.bluematt.me     → 여러 노드의 IP 주소 반환

2. DNS 서버가 현재 활성 노드들의 IP를 반환
   (DNS Seed 운영자가 주기적으로 크롤링하여 목록 갱신)

3. 반환된 IP로 직접 TCP 연결

→ DNS 인프라를 활용하므로 NAT/방화벽 뒤에서도 초기 연결이 쉬움
```

## 헷갈렸던 포인트

### Q1: 부트스트랩 노드가 중앙 서버 아닌가? 그럼 탈중앙화가 아니잖아?

**A**: 부트스트랩 노드는 **"처음 소개만 해주는 역할"**이다.
- 데이터를 저장하거나 트랜잭션을 처리하지 않음
- 한번 네트워크에 참여하면 부트스트랩 노드 없이도 완전히 독립적으로 동작
- 부트스트랩 노드가 전부 죽어도, 이미 연결된 노드들은 영향 없음
- 필요하면 아무 노드의 IP를 수동으로 입력해서 연결 가능 (`admin.addPeer()`)
- 비트코인은 DNS Seed 외에도 하드코딩된 IP 목록을 폴백으로 가지고 있음

### Q2: 20만 노드가 전부 서로 연결되어 있는 건가?

**A**: 아니다. 각 노드는 **25~50개 정도의 피어**와만 직접 연결한다.
```
[ 20만 노드의 실제 연결 구조 ]

전부 연결 (Full Mesh)     실제 구조 (Sparse Network)
   연결 수: N×(N-1)/2        연결 수: N × ~50
   = 약 200억 개              = 약 1,000만 개
   → 불가능!                  → 현실적
```
- 하지만 가십 프로토콜 덕분에 정보는 전체 네트워크에 빠르게 전파
- 네트워크 토폴로지상 어떤 두 노드 사이도 보통 4~6홉 이내로 연결됨 (Small World Network)

### Q3: 같은 트랜잭션을 중복으로 받으면 어떻게 하나?

**A**: 각 노드는 이미 받은 트랜잭션/블록의 **해시를 캐싱**해둔다.
```
1. 트랜잭션 수신 → 해시 계산
2. 이미 본 해시인가?
   → YES: 무시 (중복 전파 차단)
   → NO:  검증 후 저장 & 다른 피어에게 전파
```

### Q4: NAT/방화벽 뒤에 있는 노드는 어떻게 연결받나?

**A**: 기본적으로 **outbound 연결**은 NAT 뒤에서도 가능하다.
- 노드가 먼저 외부 노드에 연결 (outbound) → NAT 통과 가능
- 다른 노드가 나에게 연결 (inbound)하려면 → 포트포워딩 필요 (30303)
- UPnP 자동 포트포워딩을 지원하는 라우터에서는 자동으로 열림
- inbound가 안 되어도 네트워크 참여 자체는 가능 (outbound만으로 충분)

### Q5: 악의적 노드가 가짜 블록을 전파하면?

**A**: 각 노드가 **독립적으로 모든 블록을 검증**하기 때문에 무시된다.
```
악의적 노드 → 가짜 블록 전파
          ↓
수신 노드: 블록 검증
  - 해시가 올바른가?
  - 트랜잭션이 유효한가?
  - 합의 규칙을 따르는가?
  - 이전 블록과 연결되는가?
          ↓
검증 실패 → 해당 블록 폐기 + 보낸 피어 평판 하락
         → 평판이 너무 낮아지면 연결 차단
```
