# Nostr 프로토콜 — 검열 저항 탈중앙 소셜 프로토콜의 구조와 동작 원리

## 핵심 정리

### 1. Nostr란?

**Nostr**(노스터)는 **N**otes and **O**ther **S**tuff **T**ransmitted by **R**elays의 약자로, 중앙 서버 없이 **공개키 암호화 + 릴레이 서버**로 동작하는 탈중앙 소셜 프로토콜이다.

```
핵심: Nostr는 "서비스"가 아니라 "프로토콜"이다.
→ 누구든 클라이언트/릴레이를 만들 수 있음
→ 트위터처럼 하나의 회사가 통제하지 않음
```

**왜 만들어졌나:**
- 트위터(X) 같은 중앙형 SNS → 운영자가 마음대로 **계정 정지/검열** 가능
- 서버가 죽으면 **내 데이터 전부 소실**
- 알고리즘이 콘텐츠 노출을 **조작**
- Nostr는 이걸 프로토콜 레벨에서 해결

---

### 2. 핵심 구조 — 클라이언트 + 릴레이

```
┌──────────────────────────────────────────────────────┐
│                    Nostr 아키텍처                      │
│                                                       │
│  사용자 A (개인키로 서명)                               │
│  ┌──────────┐         ┌─────────┐                     │
│  │ Client   │───WS───▶│ Relay 1 │──▶ 구독자에게 전달    │
│  │ (Damus,  │───WS───▶│ Relay 2 │──▶ 구독자에게 전달    │
│  │  Amethyst│───WS───▶│ Relay 3 │──▶ 구독자에게 전달    │
│  │  등)     │         └─────────┘                     │
│  └──────────┘                                         │
│                                                       │
│  - 클라이언트: 이벤트 생성 + 서명 + 릴레이에 전송         │
│  - 릴레이: 이벤트 저장 + 구독자에게 전달 (단순 중계)       │
│  - 릴레이는 누구나 운영 가능 (WebSocket 서버)            │
│  - Relay 1이 검열해도 Relay 2, 3에 이미 전파됨          │
└──────────────────────────────────────────────────────┘
```

**클라이언트** — 사용자가 직접 사용하는 앱:
- Damus (iOS), Amethyst (Android), Snort/Iris (웹)
- 이벤트 생성, 서명, 타임라인 표시 담당
- 보통 5~10개 릴레이에 동시 연결

**릴레이(Relay)** — 이벤트를 저장하고 전달하는 WebSocket 서버:
- `wss://relay.damus.io` (무료, 범용)
- `wss://relay.nostr.band` (검색 특화)
- `wss://nos.lol` (유료, 스팸 적음)
- 릴레이마다 정책이 다름 (유료/무료, 스팸 필터 등)

### 3. 계정 = 키페어 (이메일/비번 없음)

```
┌─────────────────────────────────────────────────┐
│  계정 생성 = secp256k1 키페어 생성 (비트코인과 동일)  │
│                                                   │
│  개인키 (nsec...): 절대 공유 금지, 서명용            │
│  공개키 (npub...): 내 프로필 식별자, 공개            │
│                                                   │
│  예시:                                             │
│  npub1qe3e7... → 내 Nostr "주소" (트위터 @handle)   │
│  nsec1abc9f... → 내 "비밀번호" (잃어버리면 복구 불가)  │
│                                                   │
│  ※ 서버에 가입하는 개념이 아님                        │
│  ※ 키만 있으면 어떤 클라이언트에서든 로그인 가능        │
│  ※ 플랫폼이 계정을 정지할 수 없음 (키는 내가 보유)     │
└─────────────────────────────────────────────────┘
```

**NIP-05 — 사람이 읽을 수 있는 주소:**
```
npub1qe3e7... 은 기억하기 어려우므로
→ NIP-05 인증으로 "alice@example.com" 형태의 식별자 사용
→ 서버의 .well-known/nostr.json에 공개키를 매핑

// https://example.com/.well-known/nostr.json
{
  "names": {
    "alice": "npub1qe3e7..."
  }
}
```

### 4. 이벤트(Event) 구조 — 모든 것이 이벤트

Nostr에서 글, 프로필, 좋아요, DM 등 **모든 행위**는 "이벤트"라는 단일 JSON 구조로 표현된다.

```json
{
  "id": "sha256(직렬화된 이벤트)",
  "pubkey": "작성자 공개키 (hex)",
  "created_at": 1234567890,
  "kind": 1,
  "tags": [
    ["e", "reply-to-event-id"],
    ["p", "mention-pubkey"]
  ],
  "content": "Hello Nostr!",
  "sig": "개인키로 서명한 Schnorr 서명"
}
```

**주요 필드:**
- `id`: 이벤트 내용의 SHA-256 해시 → 위변조 감지
- `pubkey`: 작성자의 공개키 → 누가 썼는지 증명
- `kind`: 이벤트 종류 번호
- `tags`: 답글, 멘션, 해시태그 등 메타데이터
- `sig`: Schnorr 서명 (secp256k1) → 본인이 작성했음을 암호학적으로 증명

**kind 번호로 기능 구분:**

| kind | 용도 |
|------|------|
| 0 | 프로필 메타데이터 (이름, 사진, 소개) |
| 1 | 일반 글 (트윗에 해당) |
| 3 | 팔로우 목록 |
| 4 | 암호화 DM (NIP-04, 레거시) |
| 7 | 리액션 (좋아요) |
| 1059 | Gift Wrap DM (NIP-17, 개선된 암호화 DM) |
| 9735 | Zap (Lightning 결제) |
| 30023 | 장문 블로그 글 |

### 5. NIP — Nostr Implementation Possibilities

Nostr의 기능 확장은 **NIP**이라는 제안 문서로 관리된다. (비트코인의 BIP, 이더리움의 EIP와 동일한 개념)

```
주요 NIP:
- NIP-01: 기본 프로토콜 (이벤트, 구독, 릴레이 통신)
- NIP-02: 팔로우 목록
- NIP-05: DNS 기반 사람이 읽을 수 있는 식별자
- NIP-09: 이벤트 삭제 (릴레이에 삭제 요청, 강제는 아님)
- NIP-17: 암호화 DM (Gift Wrap)
- NIP-19: npub/nsec/note 등 bech32 인코딩
- NIP-57: Zap (Lightning 결제)
```

### 6. 릴레이 통신 프로토콜

클라이언트와 릴레이는 **WebSocket**으로 통신하며, JSON 배열 형태의 메시지를 주고받는다.

```
클라이언트 → 릴레이:
["EVENT", <event>]           // 이벤트 발행
["REQ", <sub_id>, <filter>]  // 이벤트 구독 (필터 조건)
["CLOSE", <sub_id>]          // 구독 해제

릴레이 → 클라이언트:
["EVENT", <sub_id>, <event>] // 매칭된 이벤트 전달
["OK", <event_id>, true]     // 이벤트 수신 확인
["EOSE", <sub_id>]           // 저장된 이벤트 전송 완료
["NOTICE", "message"]        // 에러/알림
```

**필터(Filter) 예시:**
```json
{
  "kinds": [1],
  "authors": ["pubkey1", "pubkey2"],
  "since": 1700000000,
  "limit": 50
}
// → kind 1 (일반 글) 중 특정 작성자의 최근 50개
```

### 7. Zap — 비트코인 Lightning 결제 연동

```
트위터 "좋아요" → 무료, 의미 없음
Nostr "Zap"    → 실제 비트코인(사토시) 전송

┌─────────┐    Zap 1000 sats    ┌─────────┐
│ 독자 A   │ ─────────────────▶ │ 작성자 B │
│(Lightning│                    │(Lightning│
│  Wallet) │                    │ Wallet)  │
└─────────┘                     └─────────┘

동작 흐름:
1. 작성자 B가 프로필에 Lightning 주소 등록 (lnurl 또는 Lightning Address)
2. 독자 A가 글에 Zap 버튼 클릭 → 금액 입력
3. 클라이언트가 작성자의 Lightning 서비스에 인보이스 요청
4. 독자 A의 Lightning 지갑으로 결제
5. kind 9735 이벤트로 Zap 영수증이 릴레이에 기록

- 중간 플랫폼 수수료 없음
- Lightning Network 기반 → 즉시 도착, 수수료 거의 0원
- 1 satoshi = 약 0.5원 수준 → 마이크로페이먼트 가능
```

### 8. Tor 네트워크와의 비교

| | Tor | Nostr |
|---|---|---|
| 목적 | **익명 통신** (IP 숨기기) | **검열 저항 소셜** (누구나 발언) |
| 숨기는 것 | IP 주소 (누가 접속했는지) | 없음 (공개키로 신원 공개) |
| 구조 | 어니언 라우팅 (3중 암호화 릴레이) | 릴레이 서버 (WebSocket) |
| 탈중앙 수준 | Directory Authority 의존 | 완전 탈중앙 (릴레이 자유 참여) |
| 블록체인 연관 | 노드 IP 보호용 | Lightning Network 결제 연동 |
| 속도 | 느림 (3홉 거침) | 빠름 (일반 WebSocket) |

**조합 사용:** Nostr 클라이언트를 Tor를 통해 릴레이에 연결하면, **검열 저항(Nostr) + 익명성(Tor)** 동시 달성 가능.

### 9. 트위터(X) vs Nostr 비교

| | 트위터(X) | Nostr |
|---|---|---|
| 계정 생성 | 이메일/전화번호 | 키페어 생성 (1초) |
| 계정 정지 | 운영자 재량 | **불가능** (키는 내가 보유) |
| 데이터 소유권 | 트위터 서버 | **나** (서명된 이벤트) |
| 검열 | 가능 | 릴레이 하나가 해도 다른 데서 전파 |
| 수익화 | 광고/구독 | **Zap** (비트코인 직접 수령) |
| 알고리즘 | 블랙박스 | 클라이언트가 선택 |
| 서버 비용 | 트위터 부담 | 릴레이 운영자 부담 |
| 상호운용성 | X 안에 갇힘 | 어떤 클라이언트든 호환 |

### 10. 백엔드 개발자 관점 — 릴레이 구현 시 고려사항

```
릴레이는 본질적으로 WebSocket 서버 + DB이다:

1. WebSocket 연결 관리
   - 수천 개의 동시 연결 처리
   - 구독 필터 매칭 및 이벤트 브로드캐스트

2. 이벤트 저장
   - SQLite, PostgreSQL, LMDB 등
   - kind/pubkey/created_at 인덱스 필수

3. 이벤트 검증
   - Schnorr 서명 검증 (secp256k1)
   - 이벤트 ID(SHA-256) 일치 확인

4. 스팸 방지
   - Rate Limiting
   - 유료 릴레이 (Lightning 결제로 가입)
   - PoW 요구 (NIP-13)

5. 인기 릴레이 구현체
   - strfry (C) — 고성능
   - nostream (TypeScript) — 이해하기 쉬움
   - relay (Go) — 경량
```

### 11. 한계와 과제

```
1. 키 분실 = 계정 영구 상실 (복구 방법 없음)
   → NIP-46 (Nostr Connect) 등으로 키 위임 시도 중

2. 릴레이 간 이벤트 동기화 보장 안 됨
   → 글이 안 보일 수 있음, 릴레이 선택이 중요

3. 스팸 필터링이 어려움
   → 누구나 이벤트 발행 가능, Web of Trust(NIP-32) 시도 중

4. 대규모 미디어 저장은 릴레이 부담
   → 이미지/동영상은 외부 호스팅(nostr.build 등) 사용

5. 사용자 경험(UX)이 아직 트위터에 비해 부족
   → 클라이언트 품질이 빠르게 개선 중

6. 이벤트 삭제가 "요청"일 뿐 강제가 아님
   → 릴레이가 삭제를 무시할 수 있음 (GDPR 이슈)
```

## 헷갈렸던 포인트

### Q1: Nostr는 블록체인인가?
**아니다.** Nostr는 블록체인이 아니다. 합의 메커니즘도 없고, 블록도 없고, 채굴도 없다. 그냥 **공개키 암호화 + WebSocket 릴레이** 기반의 소셜 프로토콜이다. 다만 비트코인과 같은 **secp256k1 키 체계**를 사용하고, **Lightning Network**와 연동(Zap)되기 때문에 블록체인 생태계와 밀접하다.

### Q2: 릴레이 운영자가 내 글을 검열하면?
**다른 릴레이에 이미 전파되어 있다.** 클라이언트는 보통 5~10개 릴레이에 동시에 이벤트를 전송한다. 한 릴레이가 삭제하거나 차단해도, 다른 릴레이에 같은 이벤트가 살아있다. 이것이 탈중앙의 핵심이다.

### Q3: 개인키가 유출되면?
**계정 탈취와 같다.** 개인키를 가진 사람이 곧 계정 소유자이므로, 유출되면 상대방이 내 명의로 글을 쓸 수 있다. 비밀번호 변경 같은 기능이 없다. 새 키페어를 만들고 팔로워에게 알리는 수밖에 없다. NIP-41(키 마이그레이션)이 논의 중이다.

### Q4: Nostr와 Mastodon(마스토돈)은 뭐가 다른가?
**구조가 근본적으로 다르다.** Mastodon은 **ActivityPub** 기반의 연합(federated) SNS로, 각 인스턴스 서버의 관리자가 계정을 관리한다. 서버 관리자가 검열/정지할 수 있다. Nostr는 계정이 키페어이므로 **어떤 서버도 계정을 통제할 수 없다.** Mastodon은 "여러 트위터", Nostr는 "서버 없는 트위터"에 가깝다.

### Q5: 왜 비트코인 커뮤니티에서 Nostr가 인기 있나?
**철학과 기술이 맞닿아 있다.** 1) 둘 다 **검열 저항**을 최우선 가치로 둠 2) 같은 **secp256k1** 타원곡선 암호화 사용 3) **Lightning Network** 연동으로 콘텐츠에 직접 결제(Zap) 가능 4) 둘 다 중앙 기관 없이 동작하는 프로토콜. Jack Dorsey(트위터 창업자)가 Nostr 개발에 14 BTC를 기부하기도 했다.
