# CPU, RAM, SSD, HDD — 컴퓨터 핵심 부품의 근본적 차이와 트레이드오프

## 핵심 정리

컴퓨터의 데이터 처리/저장 계층은 **"빠를수록 비싸고, 용량이 작다"**는 하나의 원칙으로 설명된다.

```
속도 빠름 ←──────────────────────────────────→ 속도 느림
용량 작음 ←──────────────────────────────────→ 용량 큼
가격 비쌈 ←──────────────────────────────────→ 가격 저렴

[CPU Register] → [L1 Cache] → [L2] → [L3] → [RAM] → [SSD] → [HDD]
   ~0.3ns          ~1ns       ~3ns   ~10ns   ~100ns   ~25μs   ~5ms

  ↑ CPU 내부 (연산 + 캐시)                 ↑ 메모리     ↑ 저장장치
```

| 구분 | CPU Cache (L1) | RAM | SSD (NVMe) | SSD (SATA) | HDD |
|------|---------------|-----|-----------|-----------|-----|
| **접근 시간** | ~1ns | ~100ns | ~25μs | ~100μs | ~5-10ms |
| **순차 읽기** | — | ~50 GB/s | ~7 GB/s | ~550 MB/s | ~150 MB/s |
| **랜덤 읽기 (4KB)** | — | ~50 GB/s | ~1 GB/s | ~200 MB/s | ~1 MB/s |
| **용량** | ~64KB | ~64GB | ~2TB | ~4TB | ~20TB |
| **가격 (GB당)** | 천문학적 | ~$3 | ~$0.08 | ~$0.06 | ~$0.02 |
| **내구성** | 무제한 | 무제한 R/W | 쓰기 횟수 제한 (TBW) | 쓰기 횟수 제한 | 기계 마모 |
| **전원 꺼도 유지?** | ❌ | ❌ 휘발성 | ✅ 비휘발성 | ✅ 비휘발성 | ✅ 비휘발성 |

## 헷갈렸던 포인트

---

### Q1: CPU는 어떻게 동작하나? — 모든 것의 시작점

**CPU는 "연산"을 하는 유일한 부품이다.** RAM, SSD, HDD는 모두 데이터를 저장만 한다. 계산은 오직 CPU만 한다.

```
[CPU가 하는 일 — Fetch-Decode-Execute 사이클]

① Fetch:   메모리에서 명령어를 가져온다
② Decode:  명령어가 뭔지 해석한다 (덧셈? 비교? 점프?)
③ Execute: 실행한다
④ 결과를 레지스터 또는 메모리에 저장
→ ①로 돌아감 (이것을 초당 수십억 번 반복)
```

**CPU 내부 구조:**

```
┌─────────────────────────────────────────────────┐
│                    CPU                           │
│  ┌──────────┐  ┌──────────┐  ┌──────────┐      │
│  │  Core 0  │  │  Core 1  │  │  Core 2  │ ...  │
│  │ ┌──────┐ │  │ ┌──────┐ │  │ ┌──────┐ │      │
│  │ │L1 64K│ │  │ │L1 64K│ │  │ │L1 64K│ │      │
│  │ │L2 256K│ │  │ │L2 256K│ │  │ │L2 256K│ │      │
│  │ │ ALU  │ │  │ │ ALU  │ │  │ │ ALU  │ │      │
│  │ │ Reg  │ │  │ │ Reg  │ │  │ │ Reg  │ │      │
│  │ └──────┘ │  │ └──────┘ │  │ └──────┘ │      │
│  └──────────┘  └──────────┘  └──────────┘      │
│  ┌──────────────────────────────────────────┐   │
│  │          L3 Cache (~30MB, 공유)           │   │
│  └──────────────────────────────────────────┘   │
└─────────────────────────────────────────────────┘
                      │ (메모리 버스)
                      ▼
              ┌───────────────┐
              │  RAM (DRAM)   │
              └───────────────┘
```

**CPU의 핵심 요소:**

| 구성 요소 | 역할 | 크기 |
|-----------|------|------|
| **Register** | 현재 연산 중인 데이터 보관 (가장 빠름) | 수십 개, 각 64비트 |
| **ALU** | 산술/논리 연산 수행 (덧셈, 비교 등) | — |
| **L1 Cache** | 가장 자주 쓰는 데이터/명령어 캐시 | ~64KB (코어별) |
| **L2 Cache** | L1에 없으면 여기서 찾음 | ~256KB (코어별) |
| **L3 Cache** | 모든 코어가 공유하는 캐시 | ~30MB |

#### CPU가 데이터를 읽는 순서 — Cache Hit/Miss

```
CPU: "변수 x의 값이 필요해"

1. Register에 있나? → 있으면 즉시 사용 (~0.3ns)
2. L1 Cache에 있나? → 있으면 사용 (~1ns)        ← Cache Hit
3. L2 Cache에 있나? → 있으면 사용 (~3ns)
4. L3 Cache에 있나? → 있으면 사용 (~10ns)
5. RAM에서 가져옴                (~100ns)        ← Cache Miss!
6. RAM에도 없으면? → Page Fault → SSD/HDD에서 로드 (~25μs ~ 5ms)
```

**Cache Miss가 성능에 미치는 영향:**

```
L1 Cache Hit:  1ns   (기준)
L3 Cache Hit:  10ns  (10배 느림)
RAM 접근:      100ns (100배 느림)
SSD 접근:      25μs  (25,000배 느림!)
HDD 접근:      5ms   (5,000,000배 느림!)

→ CPU 입장에서 RAM 접근도 "엄청 느린 것"이다
→ 캐시 효율이 성능의 핵심!
```

#### 왜 이게 개발자에게 중요한가?

```java
// ❌ 캐시 비효율적 — 열(column) 우선 순회
int[][] matrix = new int[10000][10000];
int sum = 0;
for (int col = 0; col < 10000; col++) {
    for (int row = 0; row < 10000; row++) {
        sum += matrix[row][col];  // 메모리 점프! Cache Miss 폭발
    }
}

// ✅ 캐시 효율적 — 행(row) 우선 순회
for (int row = 0; row < 10000; row++) {
    for (int col = 0; col < 10000; col++) {
        sum += matrix[row][col];  // 연속 메모리 접근 → Cache Hit!
    }
}

// 같은 결과인데 성능 차이: 5~10배!
```

이유: 배열은 메모리에 행 단위로 연속 저장된다. 행 우선 순회는 CPU 캐시 라인(64바이트)에 한 번에 올라온 데이터를 연속으로 사용하므로 Cache Hit 비율이 극적으로 높아진다.

### Q2: HDD는 왜 느린가? — 물리적 한계

**HDD는 실제로 원판(Platter)이 회전하는 기계 장치다.**

```
[HDD 내부 구조]

         ┌─── 스핀들 모터 (원판 회전)
         │
    ╔════╧════╗
    ║  ┌────┐ ║  ← 원판(Platter): 자성 물질이 코팅된 금속 디스크
    ║  │    │ ║    7200 RPM = 초당 120회 회전
    ║  └────┘ ║
    ╚════╤════╝
         │
    ─────┼───── 액추에이터 암(Actuator Arm)
         └──── 읽기/쓰기 헤드(Head): 원판 위를 이동하며 데이터 읽기
```

**데이터를 읽으려면 3단계의 물리적 이동이 필요하다:**

```
[HDD 데이터 접근 과정]

1. Seek Time (~5ms)
   헤드가 원하는 트랙(원판의 동심원)으로 이동
   → 기계 팔이 물리적으로 움직여야 한다!

2. Rotational Latency (~4ms at 7200RPM)
   원판이 회전하여 원하는 섹터가 헤드 아래로 올 때까지 대기
   → 평균적으로 반 바퀴 회전 시간

3. Transfer Time (~0.01ms)
   실제 데이터 전송 → 이건 빠르다

총 접근 시간: ~5 + ~4 + ~0.01 ≈ 9ms
```

**핵심: HDD의 병목은 "기계 부품의 물리적 이동"이다.**

그래서 HDD는:
- **순차 읽기는 괜찮다**: 헤드가 한 번 위치 잡으면 계속 읽으면 된다 (~150 MB/s)
- **랜덤 읽기는 처참하다**: 매번 헤드가 이동해야 한다 (~1 MB/s, 순차의 1/150)

```
[HDD 순차 vs 랜덤]

순차 읽기: ████████████████████ 150 MB/s (연속된 데이터)
랜덤 읽기: █                     1 MB/s (여기저기 흩어진 데이터)
```

### Q3: SSD는 왜 빠른가? — 전기 신호만으로 접근

**SSD에는 움직이는 부품이 없다.** 반도체 칩(NAND Flash)에 전기 신호로 데이터를 저장한다.

```
[SSD 내부 구조]

┌─────────────────────────────────────┐
│  Controller (두뇌)                    │
│  ┌─────┐  ┌─────┐  ┌─────┐         │
│  │NAND │  │NAND │  │NAND │  ← 플래시 메모리 칩들
│  │Flash│  │Flash│  │Flash│    (데이터 저장)
│  └─────┘  └─────┘  └─────┘         │
│  ┌──────────────────────────┐       │
│  │      DRAM Cache          │       │  ← 매핑 테이블 캐시
│  └──────────────────────────┘       │
└─────────────────────────────────────┘
```

**HDD와의 근본적 차이:**

```
HDD: "100번 트랙의 50번 섹터 읽어줘"
     → 헤드 이동(5ms) → 회전 대기(4ms) → 읽기 → 총 ~9ms

SSD: "블록 100의 페이지 50 읽어줘"
     → 전기 신호로 즉시 접근 → 총 ~25μs (NVMe)

차이: 약 360배!
```

**SSD가 랜덤 읽기에서 특히 압도적인 이유:**

```
HDD 랜덤 읽기: 매번 헤드 이동 + 회전 대기 → ~9ms/건 → 초당 ~110 IOPS
SSD 랜덤 읽기: 전기 신호 → ~25μs/건 → 초당 ~500,000 IOPS (NVMe)

차이: ~4,500배!
```

#### SATA SSD vs NVMe SSD

```
[SATA SSD]
  CPU ↔ SATA 컨트롤러 ↔ SSD
  대역폭: ~600 MB/s (SATA 3.0 한계)
  큐 깊이: 32 (동시 요청 수)

[NVMe SSD]
  CPU ↔ PCIe 버스 ↔ SSD (직통!)
  대역폭: ~8 GB/s (PCIe 4.0 x4)
  큐 깊이: 65,535 x 64K (대규모 병렬)
```

NVMe가 빠른 이유:
- **PCIe 직통**: SATA 컨트롤러 우회 → 레이턴시 감소
- **멀티 큐**: 수만 개 요청을 병렬 처리 (SATA는 32개 한계)
- **프로토콜 최적화**: NVMe는 플래시 메모리에 최적화된 프로토콜

### Q4: RAM은 왜 SSD보다도 빠른가?

**RAM(DRAM)은 커패시터(축전기)에 전하를 저장한다.**

```
[DRAM 셀 — 1비트 저장]

    워드 라인 (행 선택)
        │
    ┌───┤
    │ T │ ← 트랜지스터 (스위치)
    └───┤
        │
    ═══════ ← 커패시터 (전하 = 1, 비전하 = 0)
        │
    비트 라인 (읽기/쓰기)
```

**SSD(NAND Flash)와의 차이:**

```
DRAM:
  - 읽기: 트랜지스터 열고 전하 확인 → ~100ns
  - 쓰기: 전하 충전/방전 → ~100ns
  - 랜덤 접근: 행/열 주소 지정으로 즉시 → 어디든 같은 속도

NAND Flash (SSD):
  - 읽기: 페이지 단위(4-16KB)로 읽어야 함 → ~25μs
  - 쓰기: 블록 단위(256KB-1MB)로 지워야 쓸 수 있음 → ~250μs
  - 쓰기 증폭(Write Amplification): 1바이트 수정해도 블록 전체 재기록
```

| 특성 | DRAM (RAM) | NAND Flash (SSD) |
|------|-----------|-----------------|
| 읽기 단위 | 바이트 | 페이지 (4-16KB) |
| 쓰기 단위 | 바이트 | 블록 (256KB-1MB) |
| 접근 시간 | ~100ns | ~25μs |
| 수명 | 무제한 | 쓰기 횟수 제한 |
| 전원 유지 필요 | ✅ 필요 (휘발성) | ❌ 불필요 |
| 가격 (GB당) | ~$3 | ~$0.08 |

**RAM이 비싼 이유:** 트랜지스터 + 커패시터 = 6개 소자/셀 (SRAM) 또는 1T+1C (DRAM). NAND Flash는 1개 소자에 여러 비트(MLC=2, TLC=3, QLC=4)를 저장 → 용량당 훨씬 저렴

### Q5: 이 차이가 소프트웨어에 어떤 영향을 미치나?

#### DB가 왜 인덱스를 쓰는가 — HDD/SSD 특성 때문

```
[인덱스 없는 테이블 스캔 — HDD에서]

100만 행 검색:
  HDD: 100만 × 9ms = 9,000초 (2.5시간!) ← 랜덤 I/O
  SSD: 100만 × 25μs = 25초
  RAM: 100만 × 100ns = 0.1초

[B+Tree 인덱스 사용 — 3-4번 접근으로 찾기]

  HDD: 4 × 9ms = 36ms ← 실용적!
  SSD: 4 × 25μs = 100μs
  RAM: 4 × 100ns = 400ns
```

**B+Tree가 DB의 기본 인덱스인 이유:**
- 트리 높이가 3-4 → 디스크 접근 3-4번이면 충분
- 리프 노드가 연결 리스트 → 범위 검색 시 순차 읽기 유리 (HDD에 좋다)

#### Redis가 RAM에 데이터를 올리는 이유

```
MySQL (SSD 기반): 단순 조회 ~1ms
Redis (RAM 기반): 단순 조회 ~0.1ms (네트워크 포함)

차이: ~10배 빠름
+ Lock-free 싱글 스레드 → Context Switch 없음
+ 네트워크 I/O도 epoll로 최적화
```

#### Kafka가 HDD에서도 빠른 이유

```
Kafka의 비결: 순차 I/O만 사용!

일반 DB: 랜덤 I/O → HDD에서 ~1 MB/s
Kafka:   순차 I/O → HDD에서도 ~150 MB/s

메시지를 항상 끝에 추가(append-only) → 디스크 헤드가 이동할 일이 없다
+ OS의 Page Cache 활용 → 자주 읽는 데이터는 RAM에서 제공
+ Zero-copy (sendfile) → 커널 버퍼에서 직접 네트워크로 전송
```

### Q6: 실무에서 만나는 저장장치 관련 이슈들

| 상황 | 원인 | 해결 |
|------|------|------|
| DB 쿼리 갑자기 느려짐 | SSD 캐시(DRAM) 소진 → NAND 직접 접근 | SSD 용량 70% 이하 유지, TRIM 활성화 |
| SSD 수명 걱정 | Write Amplification + TLC/QLC 한계 | WAL 로그 분리, 임시 파일 RAM 디스크 |
| Redis 메모리 부족 | 데이터 > RAM 용량 | maxmemory-policy 설정, 클러스터 확장 |
| Kafka 디스크 풀 | 로그 세그먼트 미삭제 | retention.ms, retention.bytes 설정 |
| Docker 느려짐 | OverlayFS + 빈번한 쓰기 | 볼륨 마운트, tmpfs 활용 |

### Q7: 전체 계층 구조 — 한눈에 보기

```
┌──────────────────────────────────────────────────────┐
│ CPU Register   │ ~0.3ns │ ~수 KB   │ SRAM, 최고속     │
├──────────────────────────────────────────────────────┤
│ L1 Cache       │ ~1ns   │ ~64 KB   │ 코어별 전용       │
├──────────────────────────────────────────────────────┤
│ L2 Cache       │ ~3ns   │ ~256 KB  │ 코어별 전용       │
├──────────────────────────────────────────────────────┤
│ L3 Cache       │ ~10ns  │ ~30 MB   │ 코어 간 공유      │
├──────────────────────────────────────────────────────┤
│ RAM (DRAM)     │ ~100ns │ ~64 GB   │ 휘발성, 바이트 접근│
├──────────────────────────────────────────────────────┤
│ NVMe SSD       │ ~25μs  │ ~2 TB    │ 비휘발성, 페이지   │
├──────────────────────────────────────────────────────┤
│ SATA SSD       │ ~100μs │ ~4 TB    │ SATA 병목         │
├──────────────────────────────────────────────────────┤
│ HDD            │ ~5ms   │ ~20 TB   │ 기계식, 순차 유리  │
├──────────────────────────────────────────────────────┤
│ 테이프/콜드 스토리지│ ~수초  │ ~수 PB  │ 아카이빙 전용     │
└──────────────────────────────────────────────────────┘

위로 갈수록: 빠르고, 비싸고, 작다
아래로 갈수록: 느리고, 저렴하고, 크다
```

**트레이드오프 정리:**

```
선택 1: "전부 RAM에 올리면 되잖아!"
  → 비용 폭발 (64GB RAM ≈ $200, 64GB SSD ≈ $5)
  → 전원 꺼지면 데이터 소실

선택 2: "SSD만 쓰면 되잖아!"
  → 대용량 데이터에는 여전히 비쌈
  → 쓰기 수명 제한 (TBW)

선택 3: "HDD로 충분하지 않아?"
  → 랜덤 I/O 성능이 처참 → DB 쿼리 지연
  → 움직이는 부품 → 물리적 충격에 취약

현실적 답: 계층별로 적재적소에 배치
  핫 데이터 → RAM (Redis, Page Cache)
  웜 데이터 → SSD (DB, 파일시스템)
  콜드 데이터 → HDD/S3 (백업, 로그 아카이빙)
```
