# 커널(Kernel)이 뭔데? — 쉽게 이해하는 운영체제의 심장

## 핵심 정리

커널은 **하드웨어와 소프트웨어 사이의 통역사**다. 프로그램이 "파일 저장해줘", "네트워크로 데이터 보내줘"라고 하면, 커널이 하드웨어에게 실제 명령을 전달한다.

```
프로그램이 하고 싶은 것:       커널이 대신 해주는 것:

"파일 저장해줘"          →    디스크 컨트롤러에 명령 전달
"메모리 더 줘"           →    RAM에서 빈 공간 찾아서 할당
"네트워크로 보내줘"       →    NIC(네트워크 카드)에 패킷 전달
"화면에 그려줘"           →    GPU에 렌더링 명령 전달
```

**왜 커널이 필요한가?**

프로그램이 직접 하드웨어를 제어하면 벌어지는 일:

```
만약 커널이 없다면:

프로그램 A: "디스크 100번 섹터에 쓸게" → 쓰기 시작
프로그램 B: "디스크 100번 섹터에 쓸게" → 동시에 쓰기!
→ 데이터 파괴!

프로그램 C: "메모리 주소 0x1000 읽을게" → 다른 프로그램의 비밀번호가 있는 곳!
→ 보안 파괴!
```

커널은 **교통경찰**이다. 모든 하드웨어 접근을 중앙에서 관리하여 **충돌을 방지하고 보안을 유지**한다.

## 헷갈렸던 포인트

> 이 문서는 커널을 처음 접하는 사람도 이해할 수 있도록 이야기 형식으로 풀어간다.

### Q1: 커널은 구체적으로 어떤 일을 하나?

크게 **5가지 역할**이 있다.

#### 역할 1: 프로세스 관리 — "누가 CPU를 쓸 차례야?"

CPU는 한 번에 하나의 작업만 실행할 수 있다 (코어 1개 기준). 그런데 컴퓨터에서는 수십~수백 개의 프로그램이 동시에 돌아가는 것처럼 보인다.

```
[커널의 CPU 스케줄링]

시간 →  |--A--|--B--|--C--|--A--|--B--|--C--|
         10ms   10ms  10ms  10ms  10ms  10ms

프로그램 A, B, C를 10ms씩 번갈아 실행
→ 사람 눈에는 "동시에 실행"되는 것처럼 보임
```

커널이 하는 일:
- **프로세스 생성/종료** 관리
- **CPU 시간 배분** (스케줄링)
- 프로세스 간 **격리** (A가 B의 메모리에 접근 불가)

#### 역할 2: 메모리 관리 — "어디에 저장할까?"

```
[가상 메모리 — 커널의 마법]

프로그램 A가 보는 세상:          실제 물리 메모리(RAM):
  0x0000 ██ 내 코드               0x0000 ██ 커널 영역
  0x1000 ██ 내 데이터              0x3000 ██ A의 코드
  0x2000 ██ 내 스택               0x5000 ██ B의 코드
  (나만의 넓은 공간!)              0x7000 ██ A의 데이터
                                  0x9000 ██ B의 데이터

프로그램 B가 보는 세상:
  0x0000 ██ 내 코드         커널이 주소를 변환해서
  0x1000 ██ 내 데이터       각 프로그램이 "나만의 메모리"를
  0x2000 ██ 내 스택         가진 것처럼 착각하게 한다!
```

커널이 하는 일:
- 프로그램마다 **독립된 가상 주소 공간** 제공
- 물리 메모리(RAM)를 효율적으로 분배
- 메모리 부족 시 **디스크를 메모리처럼 사용** (Swap)

#### 역할 3: 파일 시스템 — "파일을 어떻게 저장하고 찾지?"

```
프로그램: open("/home/user/photo.jpg")
            │
            ▼
커널: "/home/user/photo.jpg"가 어디에 있지?
  → 파일 시스템(ext4) 확인 → 디스크의 어느 블록에 있는지 찾음
  → 디스크 컨트롤러에 "이 블록 읽어와" 명령
  → 데이터를 메모리에 올림
  → 프로그램에게 "여기 있어" (file descriptor 반환)
```

**"Everything is a file"** — Linux에서는 거의 모든 것이 파일이다:
```
/dev/sda        → 디스크
/dev/null       → 블랙홀 (뭘 써도 사라짐)
/proc/cpuinfo   → CPU 정보 (실제 파일이 아니라 커널이 생성)
/dev/random     → 난수 생성기
```

#### 역할 4: 디바이스 관리 — "하드웨어랑 대화하기"

프로그램이 키보드, 마우스, 디스크, 네트워크 등을 직접 제어하면 복잡하고 위험하다. 커널의 **디바이스 드라이버**가 중간에서 통역한다.

```
[프로그램] → "프린트해줘"
                │
[커널]     → 프린터 드라이버 호출
                │
[드라이버] → 프린터 하드웨어에 맞는 명령 전송
                │
[프린터]   → 출력!
```

새 하드웨어를 사면 "드라이버 설치"를 하는 이유가 이것이다. 커널이 그 하드웨어와 대화하는 방법을 알아야 한다.

#### 역할 5: 네트워크 — "인터넷 통신 처리"

```
브라우저: "google.com에 접속하고 싶어"
    │
커널:
    ├── DNS 조회 → IP 주소 획득
    ├── TCP 3-way handshake → 연결 수립
    ├── HTTP 요청 데이터를 패킷으로 쪼개기
    ├── NIC(네트워크 카드)에 전달
    └── 응답 패킷 수신 → 재조립 → 브라우저에 전달
```

### Q2: 유저 모드와 커널 모드가 뭔가? 왜 나뉘어 있나?

**핵심: 보안과 안정성 때문이다.**

```
[커널 모드 (Ring 0)] — 뭐든 할 수 있다
  ✅ 모든 메모리 접근 가능
  ✅ 하드웨어 직접 제어
  ✅ 다른 프로세스 제어
  → 여기서 버그 나면 시스템 전체가 죽는다 (블루스크린/커널 패닉)

[유저 모드 (Ring 3)] — 제한된 것만 할 수 있다
  ❌ 다른 프로세스 메모리 접근 불가
  ❌ 하드웨어 직접 제어 불가
  ❌ 커널 메모리 접근 불가
  → 여기서 버그 나면 그 프로그램만 죽는다 (Segmentation Fault)
```

**유저 모드에서 커널 기능이 필요하면? → System Call**

```
[일상생활 비유]

일반 시민(유저 모드): "건물 지어도 되나요?"
  → 직접 건설 불가. 시청(커널)에 허가 신청(System Call)해야 함

시청 공무원(커널 모드): 허가 검토 → 승인 → 건설 진행
  → 규칙에 맞는지 확인하고 대신 처리해줌
```

```
자주 쓰는 System Call:

open()   → 파일 열기
read()   → 파일/소켓에서 데이터 읽기
write()  → 파일/소켓에 데이터 쓰기
fork()   → 새 프로세스 생성
exec()   → 새 프로그램 실행
socket() → 네트워크 소켓 생성
mmap()   → 메모리 매핑
```

### Q3: Java/Spring 개발자한테 커널이 왜 중요한가?

**"내가 쓰는 모든 것이 결국 커널 위에서 동작한다."**

```
[Spring Boot 서버가 요청을 받는 과정 — 커널이 하는 일]

1. 클라이언트 TCP 연결 → 커널의 네트워크 스택이 3-way handshake 처리
2. HTTP 데이터 수신 → 커널이 소켓 버퍼에 데이터 적재
3. Tomcat이 read() → System Call로 커널에서 데이터 가져옴
4. Spring이 비즈니스 로직 실행 → JVM이 메모리 할당 요청 → 커널이 메모리 제공
5. DB 쿼리 → 커널이 네트워크 소켓으로 MySQL에 패킷 전송
6. DB 응답 → 커널이 소켓 버퍼에서 데이터 읽어옴
7. HTTP 응답 → 커널이 패킷으로 쪼개서 NIC에 전달
```

**개발하면서 마주치는 커널 관련 이슈들:**

| 에러/상황 | 커널과의 관계 |
|-----------|-------------|
| `Too many open files` | 커널이 프로세스당 열 수 있는 파일 수 제한 (ulimit) |
| `Connection refused` | 커널의 TCP 백로그 큐가 가득 참 |
| `Out of memory` | 커널의 OOM Killer가 프로세스를 강제 종료 |
| Docker 느려짐 | 커널의 cgroups 리소스 제한에 걸림 |
| `Connection timed out` | 커널의 TCP 재전송 타이머 만료 |

### Q4: 한줄 요약 — 커널이란?

```
커널 = 하드웨어와 소프트웨어 사이의 통역사 + 교통경찰 + 자원 관리자

하는 일: 프로세스 관리, 메모리 관리, 파일 시스템, 디바이스 관리, 네트워크
왜 필요: 보안, 안정성, 하드웨어 추상화 (프로그래머가 하드웨어 몰라도 코드 작성 가능)
어떻게 접근: System Call (open, read, write, fork, exec, socket, ...)
```

**더 심화된 커널 내용(Ring 구조, Monolithic vs Microkernel, 컨테이너와의 관계 등)은 [운영체제 구조와 커널 심화](kernel-and-os-structure.md)를 참고하라.**
