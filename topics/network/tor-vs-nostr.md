# Tor vs Nostr — 둘 다 탈중앙화인데 뭐가 다른 거야?

## 핵심 정리

### 한 줄 요약부터

```
Tor  = "누가 보냈는지"를 숨기는 기술 (익명 통신)
Nostr = "누가 말했는지"를 증명하는 기술 (검열 저항 SNS)

→ 해결하려는 문제 자체가 다르다!
```

### 근본적 차이 — 비유로 이해하기

```
[ Tor = 복면 쓰고 편지 보내기 ]

나(복면) → 배달부A → 배달부B → 배달부C → 받는 사람
          (출발지만 앎) (아무것도 모름) (도착지만 앎)

- 편지 내용? 아무도 못 봄 (3중 암호화)
- 누가 보냈는지? 아무도 모름
- 목적: "나"를 숨기는 것

[ Nostr = 광장에서 서명된 글을 외치기 ]

나(서명 도장 찍음) → 게시판A에 붙임
                  → 게시판B에 붙임
                  → 게시판C에 붙임

- 글 내용? 누구나 볼 수 있음 (공개)
- 누가 썼는지? 서명으로 증명됨 (공개키)
- 목적: "글"이 삭제/검열되지 않는 것
```

### 이름부터 뜯어보자

```
Tor   = The Onion Router
        "양파 라우터" — 양파 껍질처럼 암호화를 벗기며 전달
        → 라우팅(전달 경로) 기술

Nostr = Notes and Other Stuff Transmitted by Relays
        "릴레이로 전송되는 노트와 기타 것들"
        → 메시징/소셜 프로토콜
```

### 구조 비교

```
[ Tor 네트워크 구조 ]

클라이언트 ──→ Guard ──→ Middle ──→ Exit ──→ 목적지(웹사이트)

- 3개 노드를 거쳐 익명으로 인터넷에 접속
- 목적지(웹사이트)는 일반 인터넷에 있음
- 노드 운영자: 자원봉사자 ~7,000명
- 경제적 인센티브: 없음 (봉사)

[ Nostr 네트워크 구조 ]

        ┌──→ Relay A (wss://relay.damus.io)
        │
사용자 ──┼──→ Relay B (wss://nos.lol)
        │
        └──→ Relay C (wss://relay.nostr.band)

- 사용자가 "이벤트(글)"를 여러 릴레이에 동시 발행
- 릴레이 = WebSocket 서버 (JSON 저장소)
- 다른 사용자가 릴레이에서 이벤트를 구독
- 릴레이 운영자: 누구나 (개인, 기업, 커뮤니티)
- 경제적 인센티브: 유료 릴레이, 후원 등
```

### 기술 스택이 완전히 다르다

```
[ Tor — 네트워크 계층 기술 ]

계층:        응용 계층 아래 (전송 경로를 조작)
핵심 기술:   어니언 라우팅, DH 키 교환, 다중 암호화
프로토콜:    TCP 기반 (UDP 미지원)
데이터 형태: 암호화된 패킷 (바이너리)
저장:        없음 (실시간 전달만, 데이터를 저장하지 않음)

→ "어떻게 전달하느냐"에 집중

[ Nostr — 응용 계층 프로토콜 ]

계층:        응용 계층 (HTTP/WebSocket 위에서 동작)
핵심 기술:   공개키 암호화(secp256k1), 서명, JSON
프로토콜:    WebSocket (wss://)
데이터 형태: JSON 이벤트 (Event)
저장:        릴레이가 이벤트를 저장

→ "무엇을 주고받느냐"에 집중
```

### Nostr의 핵심 개념 — 이벤트(Event)

```
[ Nostr에서 모든 것은 "이벤트" ]

{
  "id": "abc123...",              // 이벤트 해시 (SHA-256)
  "pubkey": "npub1xyz...",        // 작성자 공개키
  "created_at": 1710000000,       // 타임스탬프
  "kind": 1,                     // 이벤트 종류 (1 = 텍스트 노트)
  "tags": [["p", "상대방pubkey"]], // 태그 (멘션, 답글 등)
  "content": "안녕하세요!",        // 내용
  "sig": "서명값..."              // 개인키로 서명 (위조 불가)
}

주요 kind 번호:
  kind 0  = 프로필 메타데이터 (이름, 사진, 소개)
  kind 1  = 텍스트 노트 (트윗 같은 것)
  kind 3  = 팔로우 리스트
  kind 4  = 암호화된 DM (1:1 메시지)
  kind 7  = 리액션 (좋아요, 👍)
  kind 9735 = Zap (비트코인 라이트닝 팁)

→ 계정 = 공개키/개인키 쌍 (이메일, 비밀번호 없음!)
→ 로그인 = 개인키로 서명 (비트코인 지갑과 같은 원리)
→ 누군가 글을 삭제해도 다른 릴레이에 복사본이 남아있음
```

### Nostr의 신뢰 모델 — 왜 검열 저항인가?

```
[ 트위터(X)가 글을 삭제하면? ]

나 → 트위터 서버 (1개) → 삭제됨 → 끝. 복구 불가.
     ↑ 트위터가 유일한 저장소

[ Nostr에서 릴레이가 글을 삭제하면? ]

나 → Relay A (삭제함)
   → Relay B (여전히 있음 ✅)
   → Relay C (여전히 있음 ✅)
   → 내 로컬에도 있음 ✅

→ 하나가 검열해도 다른 곳에 살아있음
→ 릴레이를 직접 띄울 수도 있음
→ 글의 진위는 "서명"으로 검증 (릴레이가 위조 불가)

[ 핵심 차이 ]
트위터: "계정"이 트위터 소유 → 정지당하면 끝
Nostr:  "계정"이 내 개인키 → 아무도 빼앗을 수 없음
        키만 있으면 어떤 릴레이에서든 활동 재개 가능
```

### 정리 비교표

| 항목 | Tor | Nostr |
|------|-----|-------|
| **한마디** | 익명 통신 네트워크 | 검열 저항 소셜 프로토콜 |
| **해결하는 문제** | "누가 접속했는지" 숨기기 | "누구도 내 글을 삭제 못하게" |
| **핵심 기술** | 어니언 라우팅 (3중 암호화) | 공개키 서명 + JSON 이벤트 |
| **네트워크 계층** | 전송 계층 (TCP 위) | 응용 계층 (WebSocket 위) |
| **노드/릴레이** | ~7,000개 릴레이 (자원봉사) | ~1,000개+ 릴레이 (자발적 운영) |
| **인센티브** | 없음 (순수 봉사) | 유료 릴레이, Zap 팁 (약간 있음) |
| **계정** | 없음 (익명이니까) | 공개키 = 계정 (비트코인과 동일) |
| **데이터 저장** | ❌ 저장 안 함 (실시간 전달) | ✅ 릴레이가 이벤트 저장 |
| **암호화** | 경로 전체 암호화 | 선택적 (DM만 암호화, 글은 공개) |
| **익명성** | ✅ 매우 높음 | ❌ 기본적으로 공개 (익명은 목적이 아님) |
| **탈중앙화 수준** | 반쯤 (DA 10개 중앙) | 높음 (중앙 권한 없음) |
| **비트코인 연관** | 노드 IP 숨기기 | Lightning 결제 (Zap) 통합 |
| **사용 예시** | 익명 웹 브라우징, .onion 사이트 | 탈중앙 트위터, 블로그, 메시징 |

### 같이 쓸 수 있나?

```
[ Tor + Nostr = 익명 + 검열 저항 ]

Nostr만:
  글은 삭제 안 되지만, 내 IP가 릴레이에 노출됨
  → "이 글을 쓴 사람이 한국 IP xxx.xxx에서 접속"

Tor + Nostr:
  나 ──Tor──→ Guard ──→ Middle ──→ Exit ──→ Nostr Relay
  → 글도 삭제 안 되고, 누가 썼는지 IP도 모름
  → 진정한 익명 + 검열 저항

실제로 Nostr 클라이언트(Amethyst 등) 중 일부는
Tor 프록시 설정을 지원함
```

```
[ 비트코인 생태계에서의 위치 ]

비트코인: "돈"의 탈중앙화      → 검열 저항 화폐
Tor:      "통신"의 익명화      → 노드 IP 보호
Nostr:    "소셜/메시징" 탈중앙화 → 검열 저항 커뮤니케이션
Lightning: "결제"의 즉시성      → 소액 결제 (Zap)

→ 이 4개가 합쳐지면:
  익명으로(Tor) 검열 없이 소통하고(Nostr)
  즉시 결제하며(Lightning) 자산을 보호(Bitcoin)하는 스택
```

## 헷갈렸던 포인트

### Q1: Nostr도 릴레이 운영자가 검열할 수 있지 않나?

**A**: 맞다. 개별 릴레이는 검열할 수 있다. 하지만 **시스템 전체**는 검열이 안 된다.

```
트위터가 검열하면:
  → 글이 사라짐. 끝. 대안 없음.

Nostr 릴레이 하나가 검열하면:
  → 다른 릴레이 10개에 같은 글이 있음
  → 새 릴레이를 직접 띄울 수도 있음 (1시간이면 가능)
  → 글에 서명이 있으니 위조/변조도 불가

비유:
  트위터 = 신문사 1곳이 기사를 삭제 → 없어짐
  Nostr  = 전단지를 100곳에 붙임 → 1곳이 떼어도 99곳에 남아있음
```

### Q2: Nostr 계정이 해킹되면 어떡하나? 복구할 수 있나?

**A**: **복구 불가**. 이게 Nostr의 가장 큰 약점이다.

```
트위터: 비밀번호 분실 → 이메일로 재설정 (중앙 서버가 관리)
Nostr:  개인키 유출   → 끝. 복구 방법 없음.

개인키(nsec) = 계정 그 자체
  → 유출되면 공격자가 내 이름으로 글 작성 가능
  → "계정 정지" 같은 개념이 없음 (중앙 관리자가 없으니까)

대응 방법:
  → NIP-46 (Remote Signing): 개인키를 직접 노출하지 않고 서명
  → NIP-26 (Delegation): 위임 키 사용
  → 하드웨어 서명기 연동 (실험적)
  → 결국 비트코인 개인키 관리와 같은 문제
     "not your keys, not your account"
```

### Q3: Nostr가 트위터를 대체할 수 있나?

**A**: 프로토콜 수준에서는 가능하지만, 현실적으로는 아직 멀었다.

```
Nostr의 강점:
  ✅ 계정 정지/검열 불가
  ✅ 플랫폼 종속 없음 (클라이언트 자유롭게 선택)
  ✅ 비트코인 Lightning 결제 통합
  ✅ 개발자 친화적 (단순한 프로토콜)

Nostr의 현실적 한계:
  ❌ 스팸 필터링 어려움 (중앙 관리자가 없으니까)
  ❌ 콘텐츠 검색/추천 알고리즘 부족
  ❌ 일반 사용자에게 "개인키 관리"는 너무 어려움
  ❌ 릴레이 유지 비용 (누가 서버비를 내나?)
  ❌ 사용자 수: Nostr ~수십만 vs 트위터 ~5억
```

### Q4: Tor랑 Nostr를 왜 헷갈리는 거야? 공통점이 있나?

**A**: 둘 다 "중앙 서버 없이 동작"하려는 철학이 같아서 헷갈린다.

```
공통점:
  ✅ 중앙 서버/회사에 의존하지 않음
  ✅ 누구나 노드/릴레이를 운영할 수 있음
  ✅ 검열에 저항하려는 목적
  ✅ 오픈소스
  ✅ 비트코인 생태계와 밀접

결정적 차이:
  Tor  → "통신 경로"를 숨김 → 네트워크 인프라 기술
  Nostr → "콘텐츠"를 보존   → 애플리케이션 프로토콜

쉽게 말하면:
  Tor  = 도로를 달리는 차의 번호판을 가리는 기술
  Nostr = 게시판에 붙인 글이 절대 지워지지 않게 하는 기술
  → 하는 일이 완전히 다름!
```

### Q5: bitchat은 서버가 클라우드에 배포되어 있는 거 아닌가?

**A**: **아니다.** bitchat에는 "우리 서버"라는 게 없다. 이게 핵심이다.

```
[ 일반 채팅 앱 (카카오톡, 텔레그램) ]

나 ──→ 카카오 서버 (AWS/GCP에 배포) ──→ 상대방
       ↑
       중앙 서버가 메시지를 저장, 전달, 관리
       회사가 서버비를 내고, 회사가 통제권을 가짐

[ bitchat — 중앙 서버 없음 ]

방법 1: 블루투스 메시 (오프라인)
  나 ──Bluetooth──→ 근처 사람A ──→ 사람B ──→ ... ──→ 상대방
                   (최대 7홉)
  → 인터넷 자체가 필요 없음!
  → 서버? 없음. 근처 사람들 폰이 중계기 역할

방법 2: Nostr 릴레이 (온라인)
  나 ──→ Relay A (누군가 자원봉사로 운영)
     ──→ Relay B (다른 사람이 운영)
     ──→ ... (290개+ 릴레이에 분산)
  → "bitchat 서버"가 아님!
  → 전 세계 Nostr 릴레이를 그냥 빌려 쓰는 것
  → 릴레이 하나가 죽어도 다른 289개가 살아있음
```

```
[ bitchat 아키텍처 — 서버가 없다는 게 무슨 뜻인가 ]

카카오톡:
  서버 위치: 카카오 데이터센터 (한국)
  서버 비용: 카카오가 부담
  서버 죽으면: 채팅 불가
  검열 가능: ✅ 카카오가 통제

텔레그램:
  서버 위치: 두바이 등 (분산 데이터센터)
  서버 비용: 텔레그램이 부담
  서버 죽으면: 채팅 불가
  검열 가능: ✅ 텔레그램이 통제

bitchat:
  서버 위치: "서버"가 없음
  서버 비용: 릴레이 운영자들이 각자 부담
  릴레이 죽으면: 다른 릴레이로 자동 전환 + 블루투스 폴백
  검열 가능: ❌ 누구도 전체를 통제할 수 없음

→ "클라우드에 배포된 서버"라는 개념 자체가 없다!
→ 앱 코드는 오픈소스 (Swift, 98.5%)
→ 코드 자체가 "어떤 Nostr 릴레이에든 연결"하도록 설계
```

```
[ 그래서 Tor는 어디에 쓰이는 건가? ]

bitchat 자체는 Tor를 쓰지 않음.

하지만 bitchat이 Nostr 릴레이에 연결할 때:
  내 폰 ──WebSocket──→ Nostr Relay
  → 릴레이는 내 IP를 알 수 있음

여기에 Tor를 끼우면:
  내 폰 ──Tor──→ Guard ──→ Middle ──→ Exit ──→ Nostr Relay
  → 릴레이가 내 IP를 모름
  → Tor = 네트워크 경로를 숨기는 "투명 망토"
  → bitchat = 메시지를 주고받는 "앱"
  → 서로 다른 계층에서 동작

정리:
  Tor   = 네트워크 계층 (L3~L4) — "어떻게 전달하느냐"
  Nostr = 응용 계층 (L7)        — "무엇을 주고받느냐"
  bitchat = Nostr 위의 앱       — "사용자가 실제로 쓰는 것"
```
