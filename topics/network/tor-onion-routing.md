# Tor와 어니언 라우팅 — 익명 통신의 원리부터 블록체인 활용까지

## 핵심 정리

### Tor란?

Tor(The Onion Router)는 **인터넷 통신을 익명화**하는 네트워크다. VPN이 "한 곳을 경유"하는 거라면, Tor는 **3개의 중계 노드를 거치면서 양파 껍질처럼 암호화를 겹겹이 벗기는** 방식이다.

```
[ VPN vs Tor 비교 ]

VPN:
  나 ──암호화──→ VPN 서버 ──→ 목적지
                   │
        VPN 업체가 내 트래픽을 "볼 수 있음"
        VPN 업체를 신뢰해야 함

Tor:
  나 ──3중 암호화──→ Guard ──2중 암호화──→ Middle ──1중 암호화──→ Exit ──→ 목적지
                     │                     │                    │
              내 IP만 앎            앞뒤 노드만 앎          목적지만 앎
              목적지 모름           나도 모르고              내 IP 모름
                                  목적지도 모름

→ 어떤 단일 노드도 "누가 어디에 접속하는지" 전체 그림을 알 수 없음
```

### 어니언 라우팅 (Onion Routing) — 핵심 원리

"어니언"이라는 이름은 **암호화를 양파 껍질처럼 겹겹이 감싸고, 중계 노드마다 한 겹씩 벗기는** 구조에서 나왔다.

```
[ 어니언 라우팅 — 암호화 과정 ]

1단계: 클라이언트가 경로(Circuit)를 선택
  Guard Node → Middle Node → Exit Node
  (3개 노드를 랜덤으로 선택)

2단계: 각 노드의 공개키를 Tor Directory에서 받아옴

3단계: 메시지를 3중으로 암호화 (안에서 바깥으로)

  원본 데이터: "GET /index.html"

  1차 암호화 (Exit 노드의 키로):
  ┌──────────────────────────────────┐
  │ Encrypt_Exit("GET /index.html") │
  └──────────────────────────────────┘

  2차 암호화 (Middle 노드의 키로):
  ┌────────────────────────────────────────┐
  │ Encrypt_Middle(                        │
  │   Encrypt_Exit("GET /index.html")      │
  │ )                                      │
  └────────────────────────────────────────┘

  3차 암호화 (Guard 노드의 키로):
  ┌──────────────────────────────────────────────┐
  │ Encrypt_Guard(                               │
  │   Encrypt_Middle(                            │
  │     Encrypt_Exit("GET /index.html")          │
  │   )                                          │
  │ )                                            │
  └──────────────────────────────────────────────┘
  → 이것이 "양파" — 3겹으로 감싸진 암호문
```

```
[ 어니언 라우팅 — 복호화 과정 (각 노드가 한 겹씩 벗김) ]

클라이언트 → Guard Node:
  ┌──────────────────────────────────────────────┐
  │ 🧅🧅🧅 3겹 암호화된 패킷                       │
  └──────────────────────────────────────────────┘
  Guard: 자기 키로 한 겹 벗김 → "Middle에게 보내라"
         나는 클라이언트 IP를 알지만, 목적지는 모름

Guard → Middle Node:
  ┌────────────────────────────────────────┐
  │ 🧅🧅 2겹 암호화된 패킷                   │
  └────────────────────────────────────────┘
  Middle: 자기 키로 한 겹 벗김 → "Exit에게 보내라"
          나는 Guard와 Exit만 알지, 클라이언트도 목적지도 모름

Middle → Exit Node:
  ┌──────────────────────────────────┐
  │ 🧅 1겹 암호화된 패킷               │
  └──────────────────────────────────┘
  Exit: 자기 키로 마지막 겹 벗김 → 원본 데이터 노출
        나는 목적지를 알지만, 클라이언트 IP는 모름

Exit → 목적지 서버:
  "GET /index.html" (평문 또는 HTTPS)

→ 응답도 역순으로 다시 3중 암호화되어 돌아옴
```

### Tor 네트워크 구성 요소

```
[ Tor 네트워크 구조 ]

┌──────────────────────────────────────────────────────────┐
│                   Tor Network                            │
│                                                          │
│  ┌───────────┐    ┌───────────┐    ┌───────────┐        │
│  │  Guard     │    │  Middle   │    │   Exit    │        │
│  │  (Entry)   │───→│  Relay    │───→│  Relay    │───→ 인터넷
│  │  Node      │    │           │    │           │        │
│  └───────────┘    └───────────┘    └───────────┘        │
│       ↑                                                  │
│  클라이언트                                                │
│                                                          │
│  ┌─────────────────┐                                     │
│  │ Directory       │  ← 전체 릴레이 목록, 공개키,           │
│  │ Authority (DA)  │    대역폭, 가동시간 등 메타정보 관리     │
│  │ (10개, 하드코딩) │                                     │
│  └─────────────────┘                                     │
│                                                          │
│  ┌─────────────────┐                                     │
│  │ Hidden Service  │  ← .onion 주소로만 접근 가능           │
│  │ (.onion)        │    Exit 노드 없이 Tor 내부에서 통신    │
│  └─────────────────┘                                     │
└──────────────────────────────────────────────────────────┘
```

**각 구성 요소의 역할**:

| 구성 요소 | 역할 | 특징 |
|-----------|------|------|
| Guard (Entry) Node | 클라이언트가 최초로 연결하는 노드 | 클라이언트 IP를 아는 유일한 노드. 2~3개월 고정 사용 (Guard 고정 정책) |
| Middle Relay | 중간 경유지 | 앞뒤 노드만 알고, 출발지·목적지 모두 모름 |
| Exit Relay | Tor 네트워크에서 일반 인터넷으로 나가는 출구 | 목적지를 아는 유일한 노드. 운영 리스크 높음 (트래픽 책임 논란) |
| Directory Authority | 릴레이 목록과 메타데이터 관리 | 10개 조직이 운영, 투표로 합의. Tor의 "전화번호부" |
| Hidden Service | .onion 주소를 가진 서버 | 서버 위치도 익명. Exit 없이 Tor 내부에서만 통신 |

### Circuit 구축 과정 — 텔레스코핑 (Telescoping)

Tor는 한 번에 3중 터널을 만드는 게 아니라, **한 홉씩 확장**한다.

```
[ Circuit 구축: Telescoping 방식 ]

1단계: Guard 노드와 TLS + Diffie-Hellman 키 교환
  클라이언트 ←─ 암호화 채널 ──→ Guard
  (공유 비밀키 K1 생성)

2단계: Guard를 통해 Middle 노드와 키 교환
  클라이언트 ─── K1로 암호화 ───→ Guard ──→ Middle
  (Guard를 터널로 사용하여 Middle과 K2 생성)
  → Guard는 K2 내용을 볼 수 없음 (end-to-end 암호화)

3단계: Guard+Middle을 통해 Exit 노드와 키 교환
  클라이언트 ─── K1(K2(...)) ───→ Guard ──→ Middle ──→ Exit
  (K3 생성. Guard도 Middle도 K3 내용을 볼 수 없음)

결과: 3개의 독립적인 대칭키 (K1, K2, K3) 확보
  → 메시지를 보낼 때 K3 → K2 → K1 순서로 암호화
  → 각 노드가 자기 키로 한 겹만 벗김

[ 왜 한번에 안 하고 한 홉씩? ]
  → 각 구간의 키를 독립적으로 협상
  → 중간 노드가 다른 구간의 키를 절대 알 수 없음
  → 한 노드가 손상되어도 다른 구간은 안전
```

### Hidden Service (.onion) — 서버도 익명으로 운영

일반 Tor 사용은 "클라이언트를 익명화"하는 거지만, **.onion은 서버 자체도 익명화**한다.

```
[ Hidden Service (.onion) 동작 원리 ]

1. Hidden Service가 자신의 Circuit을 만들어서
   "Introduction Point" (소개 지점)에 연결
   → Directory에 "나한테 연락하려면 이 Introduction Point로 와"라고 등록

2. 클라이언트가 .onion 주소로 접속하려 할 때:
   → Directory에서 Introduction Point 정보를 받음
   → "Rendezvous Point" (만남 지점)를 랜덤으로 선택

3. 클라이언트 → Introduction Point → Hidden Service에게:
   "Rendezvous Point에서 만나자"

4. Hidden Service가 자기 Circuit으로 Rendezvous Point에 연결

5. 클라이언트와 Hidden Service가 Rendezvous Point에서 만남
   → 양쪽 다 3홉 Circuit을 통해 연결
   → 총 6홉 경유! (클라이언트 3홉 + 서버 3홉)

클라이언트 → Guard → Middle → Rendezvous ← Middle ← Guard ← Hidden Service
                     Point

→ 서버 IP도 클라이언트 IP도 서로 모름
→ .onion 주소 = 서버의 공개키 해시
  (v3: 56자, 예: vww6ybal4bd7szmgncyruucpgfkqahz...wyd.onion)
```

### Tor vs VPN vs 프록시 — 뭐가 다른 건가?

```
[ 익명성 수준 비교 ]

일반 접속:     나 ──────────────────────→ 목적지
               ISP, 목적지 모두 내 IP를 알고 내용도 볼 수 있음

프록시:        나 ──→ 프록시 서버 ──→ 목적지
               프록시가 모든 걸 알고, 암호화 없을 수 있음

VPN:           나 ══암호화══→ VPN 서버 ──→ 목적지
               VPN 업체가 모든 걸 알 수 있음 (신뢰 필요)
               but ISP는 VPN 사용 여부만 알고 내용은 모름

Tor:           나 ══→ Guard ══→ Middle ══→ Exit ──→ 목적지
               어떤 단일 주체도 전체 그림을 모름 (신뢰 불필요)
```

| 항목 | 프록시 | VPN | Tor |
|------|--------|-----|-----|
| 암호화 | 없거나 약함 | 강력 (1겹) | 3겹 (어니언) |
| 경유 노드 | 1개 | 1개 | 3개 |
| 신뢰 모델 | 프록시 운영자 신뢰 | VPN 업체 신뢰 | **신뢰 불필요** |
| 속도 | 빠름 | 약간 느림 | **느림** (3홉) |
| 익명성 | 낮음 | 중간 | **높음** |
| IP 숨김 | 목적지로부터만 | 목적지로부터만 | **모든 경유지로부터** |
| 비용 | 무료~유료 | 유료 | **무료** |
| 차단 난이도 | 쉬움 | 보통 | 어려움 (Bridge/Pluggable Transport) |

### 비트코인이 Tor를 활용하는 방식

비트코인 노드의 약 10%가 Tor 네트워크 위에서 운영된다.

```
[ 왜 비트코인 노드를 Tor로 운영하나? ]

문제: 비트코인 P2P는 원래 평문 통신 (BIP 324 이전)
  → ISP가 "이 IP가 비트코인 노드를 운영중"이라는 걸 알 수 있음
  → 특정 노드가 특정 트랜잭션을 "최초로 전파"했는지 추적 가능
  → 정부가 비트코인 노드를 차단하거나 감시 가능

Tor 사용 시:
  → 노드의 실제 IP가 네트워크에 노출되지 않음
  → 트랜잭션 출처 추적 불가
  → 국가 수준 검열 우회 가능

[ Bitcoin Core의 Tor 설정 ]

# bitcoin.conf
proxy=127.0.0.1:9050        # Tor SOCKS5 프록시 사용
listen=1                     # .onion 주소로 inbound 연결 받기
onlynet=onion                # Tor 연결만 허용 (clearnet 차단)

→ Bitcoin Core가 자동으로 .onion hidden service 생성
→ 다른 Tor 노드가 이 .onion 주소로 연결 가능
→ addrv2 (BIP 155) 메시지로 .onion 주소를 다른 피어에게 공유
```

```
[ Tor 비트코인 노드의 연결 구조 ]

Clearnet 노드:                      Tor 노드:
  IP: 82.196.7.109:8333              .onion: abc...xyz.onion:8333
  → 누구나 IP를 알 수 있음             → 실제 IP를 아무도 모름

Clearnet 노드 ↔ Clearnet 노드:  직접 TCP 연결
Tor 노드 ↔ Tor 노드:            Tor Circuit을 통한 연결
Clearnet ↔ Tor:                  Tor 프록시를 통해 연결 가능

비트코인 네트워크 전체:
  ┌─────────────────────────────────────────┐
  │          Bitcoin P2P Network             │
  │                                         │
  │   Clearnet   ←─ 양방향 ──→  Tor         │
  │   (~90%)     연결 가능      (~10%)       │
  │   IP 노출                  IP 숨김       │
  │                                         │
  │          I2P, CJDNS도 지원 (소수)        │
  └─────────────────────────────────────────┘
```

### Tor의 약점과 공격 벡터

Tor는 강력한 익명성을 제공하지만, 완벽하지는 않다.

```
[ 주요 공격 방식 ]

1. 트래픽 상관관계 분석 (Traffic Correlation Attack)
   ─────────────────────────────────────────────
   공격자가 Guard 노드와 Exit 노드를 "동시에" 관찰하면:

   Guard 쪽: "14:00:00.123에 500KB 들어옴"
   Exit 쪽:  "14:00:00.456에 500KB 나감"
   → 타이밍과 크기가 일치 → 같은 사용자로 연결 가능!

   NSA 같은 글로벌 감시자가 인터넷 백본을 관찰하면 가능
   → Tor의 가장 근본적인 한계
   → 대응: padding (더미 트래픽), 타이밍 지연 등 연구 중

2. 악의적 Exit 노드
   ─────────────────────────────────────────────
   Exit 노드는 평문 트래픽을 볼 수 있음 (HTTP인 경우)
   → HTTPS를 사용하면 Exit 노드도 내용을 볼 수 없음
   → 항상 HTTPS 사용이 중요한 이유

3. 악의적 Guard + Exit 동시 운영 (Sybil Attack)
   ─────────────────────────────────────────────
   공격자가 다수의 릴레이를 운영하여
   Guard와 Exit에 동시에 선택될 확률을 높임
   → Directory Authority의 릴레이 검증으로 완화
   → Guard 고정 정책 (2~3개월 같은 Guard 사용)으로 노출 빈도 감소

4. 비트코인 특화 공격: 트랜잭션 출처 추적
   ─────────────────────────────────────────────
   악의적 Tor Exit 노드가 비트코인 트랜잭션을 관찰:
   → 어떤 .onion 노드가 특정 Tx를 "최초로 전파"했는지 추적
   → 대응: Dandelion++ (BIP 156) — Tx를 먼저 소수 경로로 몰래 전달 후
           랜덤 시점에 전체 네트워크에 공개 (민들레 홀씨처럼)

   Dandelion++ 동작:
   [ Stem Phase (줄기) ]        [ Fluff Phase (홀씨) ]
   A → B → C → D               D → 전체 네트워크에 가십
   (1:1로 몰래 전달)             (여기서부터 일반 전파)
   → 관찰자는 D가 출처라고 생각   → 실제 출처 A는 숨겨짐
```

### Tor 네트워크 현황 (2024~2025)

```
[ Tor 네트워크 규모 ]

릴레이 노드:         ~7,000개
  ├─ Guard 자격:     ~3,000개 (Stable + Fast 플래그)
  ├─ Exit 릴레이:    ~1,500개
  └─ Middle 릴레이:  나머지

Bridge 노드:         ~2,500개 (비공개, 검열 우회용)
일일 사용자:         ~200~300만 명
총 대역폭:          ~600 Gbps

Directory Authority:  10개 (하드코딩)
  - 조직별 분산: EFF, Torproject, 개인 운영자 등
  - 합의 문서를 1시간마다 투표로 갱신
```

## 헷갈렸던 포인트

### Q1: 결국 Tor는 탈중앙화인 거야? 누군가가 노드를 운영하는 거야?

**A**: **반쯤 탈중앙화**다. "누가 운영하냐"부터 정리하면:

```
[ Tor 노드는 누가 운영하는가? ]

정답: "아무나". 지금 이 글 읽는 사람도 운영할 수 있다.

┌────────────────────────────────────────────────────────────┐
│  Tor 릴레이 운영자 = 전 세계 자원봉사자 + 단체 + 기업      │
│                                                            │
│  ☑ 프라이버시 활동가가 자기 서버에 띄운 Guard 노드          │
│  ☑ 독일 대학교 연구실에서 돌리는 Middle 릴레이              │
│  ☑ EFF(전자프론티어재단)가 후원하는 Exit 노드               │
│  ☑ 취미로 라즈베리파이에 올린 개인 릴레이                   │
│  ☑ 기업(CloudFlare 등)이 실험적으로 운영하는 노드           │
│                                                            │
│  → 중앙 회사가 "고용"하는 게 아님                          │
│  → 누구든 Tor 소프트웨어를 설치하면 노드가 됨              │
│  → 현재 ~7,000개 릴레이가 전 세계에 분산 운영 중           │
└────────────────────────────────────────────────────────────┘

[ 실제로 노드 띄우는 건 이렇게 쉬움 ]
$ apt install tor
$ vi /etc/tor/torrc
  ORPort 9001            # 릴레이 포트 열기
  Nickname MyRelay       # 릴레이 이름
  ContactInfo me@email   # 연락처
  ExitPolicy reject *:*  # Exit 안 할 거면 이렇게
$ systemctl start tor
→ 끝. 이제 이 서버가 Tor 릴레이 중 하나가 됨
```

```
[ 그래서 탈중앙화인가? ]

완전 탈중앙화 (비트코인처럼):  ❌ 아님
완전 중앙화 (VPN 회사처럼):   ❌ 아님
하이브리드 (반쯤 탈중앙화):   ✅ 이거

┌─────────────────────────────────────────────────────────┐
│                                                         │
│    탈중앙화인 부분 (노드 운영)                           │
│    ──────────────────────────                           │
│    - 릴레이 노드 ~7,000개: 누구나 자유롭게 참여/탈퇴    │
│    - 허가 필요 없음. 검열도 없음                        │
│    - 특정 노드가 죽어도 네트워크는 계속 동작             │
│    - 비트코인 채굴자처럼 "자발적 참여"                  │
│                                                         │
│    중앙화된 부분 (Directory Authority)                   │
│    ──────────────────────────                           │
│    - DA 10개: Tor 프로젝트가 하드코딩으로 지정           │
│    - 새 DA를 추가하려면 Tor 소프트웨어 업데이트 필요     │
│    - "어떤 릴레이가 신뢰할 만한가" 투표로 결정           │
│    - 이 10개가 전부 죽으면 새 노드가 네트워크에 못 들어옴│
│                                                         │
│    비유:                                                │
│    ┌─────────────────────────────────────────────┐      │
│    │ Tor = "운동장은 누구나 쓸 수 있지만,         │      │
│    │        운동장 관리인은 10명이 지정되어 있다"  │      │
│    │                                             │      │
│    │ 비트코인 = "운동장도, 관리도 전부 참여자가   │      │
│    │            알아서 한다"                      │      │
│    │                                             │      │
│    │ VPN = "운동장 주인이 1명이고,                │      │
│    │        그 사람이 전부 통제한다"              │      │
│    └─────────────────────────────────────────────┘      │
└─────────────────────────────────────────────────────────┘
```

```
[ 왜 완전 탈중앙화 안 했는가? ]

비트코인처럼 완전 탈중앙화하려면:
  → 악의적 노드를 걸러내는 메커니즘이 필요
  → 비트코인은 "채굴에 전기를 태우는 비용(PoW)"으로 해결
  → Tor는 "릴레이 운영"에 경제적 인센티브가 없음 (무료 봉사!)

  Tor 노드 운영자에게 돌아오는 보상:
    💰 금전적 보상?     없음 (서버비는 자기 돈)
    🏆 명예?            Tor Metrics에 이름이 뜸 (그게 다)
    🌍 사회적 가치?     "인터넷 자유에 기여"라는 만족감

  → 경제적 인센티브 없이 Sybil 공격을 막으려면
    "신뢰할 수 있는 심판"이 필요
  → 그게 Directory Authority의 역할
  → 완전한 탈중앙화와 보안 사이의 현실적 타협
```

```
[ 요약: VPN vs Tor vs 비트코인 — 신뢰 모델 비교 ]

VPN:
  "NordVPN 회사를 믿어야 한다"
  → 그 회사가 로그를 안 남긴다고 "약속"만 함
  → 회사가 거짓말하면? 끝.

Tor:
  "아무도 안 믿어도 된다" (단, DA 10개는 반쯤 믿어야 함)
  → 7,000개 노드 중 어느 하나를 못 믿어도 OK
  → 3개 노드가 동시에 공모하지 않는 한 안전
  → DA는 "릴레이 목록"만 관리, 트래픽은 안 봄

비트코인:
  "정말로 아무도 안 믿어도 된다"
  → 수학(SHA-256)과 경제적 인센티브(채굴 보상)로 보장
  → 사람을 믿는 게 아니라 시스템을 믿는 구조
```

### Q2: (심화) Directory Authority 10개가 전부 죽으면 어떻게 되나?

**A**: 맞다, DA는 Tor의 가장 큰 중앙화 포인트다.
```
하지만:
  - 10개 조직이 독립적으로 운영 (단일 주체 아님)
  - 과반수(6/10) 합의가 필요 → 5개까지 매수해도 소용없음
  - DA가 하는 일은 "릴레이 목록 제공"뿐
  - DA가 전부 죽어도, 캐싱된 합의 문서로 당분간 동작
  - 비트코인의 DNS Seed와 비슷한 역할

비트코인 DNS Seed:  6~8명 운영, 노드 목록 제공
Tor DA:             10개 운영, 릴레이 목록 제공
→ 둘 다 "초기 연결 도움"이라는 점에서 구조적으로 유사
```

### Q3: Tor를 쓰면 비트코인 거래가 완전히 익명인가?

**A**: 네트워크 수준 익명성은 높아지지만, **온체인 분석**은 별개 문제다.
```
Tor가 숨겨주는 것:
  ✅ 노드의 실제 IP 주소
  ✅ 어떤 노드가 트랜잭션을 최초 전파했는지
  ✅ ISP/정부의 비트코인 트래픽 감시

Tor가 숨겨주지 못하는 것:
  ❌ 블록체인에 기록된 트랜잭션 내역 (영구 공개)
  ❌ 주소 간 자금 흐름 (Chainalysis 등으로 추적 가능)
  ❌ 거래소 KYC로 연결된 주소 정보

→ 완전한 프라이버시를 위해서는
  Tor + CoinJoin(트랜잭션 믹싱) + 주소 재사용 금지
  또는 프라이버시 코인(Monero, Zcash)이 필요
```

### Q4: VPN + Tor를 같이 쓰면 더 안전한가?

**A**: 상황에 따라 다르고, 오히려 해로울 수 있다.
```
구성 1: VPN → Tor (VPN 먼저, 그 위에 Tor)
  나 ──→ VPN ──→ Guard ──→ Middle ──→ Exit ──→ 목적지
  장점: ISP가 Tor 사용 자체를 모름 (VPN 트래픽으로 보임)
  단점: VPN 업체가 "이 사람이 Tor를 쓴다"는 걸 앎

구성 2: Tor → VPN (Tor 먼저, 그 위에 VPN)
  나 ──→ Guard ──→ Middle ──→ Exit ──→ VPN ──→ 목적지
  장점: Exit 노드의 평문 트래픽 감시 차단
  단점: VPN 업체가 고정 출구 → 트래픽 상관관계 분석에 취약

Tor 프로젝트 공식 입장:
  "대부분의 경우 VPN 추가는 익명성을 높이지 않는다.
   오히려 공격 표면만 늘릴 수 있다."
→ 목적에 따라 판단해야 함 (검열 우회 목적이면 구성 1이 유용)
```

### Q5: Guard 노드를 왜 2~3개월 고정하나? 매번 바꾸는 게 더 안전하지 않나?

**A**: 직관과 반대로, **자주 바꾸면 더 위험**하다.
```
매번 랜덤으로 Guard를 선택한다면:
  → 100번 연결하면, 악의적 Guard에 걸릴 확률이 누적됨
  → 악의적 Guard + 악의적 Exit에 동시에 걸리면 익명성 파괴
  → 시간이 지날수록 "언젠가 걸림"

Guard를 고정하면:
  → 운 좋게 정직한 Guard를 골랐으면 → 계속 안전
  → 운 나쁘게 악의적 Guard를 골랐으면 → 위험하지만...
  → "일부 사용자가 항상 위험"보다
     "모든 사용자가 가끔씩 위험"이 더 나쁨
  → 고정하면 위험 노출이 소수 사용자에게만 제한됨
```

### Q6: 이더리움 노드도 Tor를 쓸 수 있나?

**A**: 가능하지만, 비트코인만큼 잘 지원되지는 않는다.
```
비트코인:
  - Bitcoin Core에 Tor 프록시 설정 내장
  - .onion hidden service 자동 생성
  - addrv2로 .onion 주소 P2P 공유
  - 전체 노드의 ~10%가 Tor

이더리움:
  - Geth에서 --torproxy 옵션 지원 (실험적)
  - 하지만 UDP 기반 노드 탐색(discv4)이 Tor와 호환 안 됨
    → Tor는 TCP만 지원, UDP 불가
    → 노드 탐색을 별도로 해결해야 함
  - 실제 Tor 이더리움 노드는 매우 적음

→ 비트코인이 Tor 친화적인 이유:
  모든 통신이 TCP라서 Tor와 자연스럽게 호환
```
